\begin{abstract}
	Temporal information plays a crucial role in many database applications, however support for queries on such data is limited.
	We present an index structure to support \emph{range-duration queries} over temporal intervals constraining both the \emph{range}, i.e. the position along the timeline, and the \emph{duration} of the interval.
	Our data structure features provable bounds on the running time of all the operations, allows for a simple implementation, and sports very predictable query performance.
	We benchmark our implementation on a variety of datasets and query workloads, investigating both the query rate and the behavior of the individual queries.
	Our experimental evaluation shows that, compared to other data structures, our index indeed better supports range-duration queries.
	While other data structures are faster on specialized workloads constraining only the duration or only the range, overall our data structure expresses the best compromise across all tested workloads.
\end{abstract}

\section{Introduction}

Temporal information plays a crucial role in many database applications: many database management systems provide indeed automated version control of the data and \emph{time travel} facilities, allowing to access past history efficiently.
Past research concentrated on the important operators of temporal aggregation~\cite{DBLP:conf/icde/KlineS95,DBLP:journals/tkde/MoonLI03,DBLP:conf/edbt/BohlenGJ06,DBLP:conf/ssd/PiatovH17}, temporal joins~\cite{DBLP:conf/icde/PiatovHD16}, and time travel~\cite{DBLP:conf/sigmod/KaufmannMVFKFM13}.

All these approaches focus on handling temporal intervals, but concentrate on the \emph{position} of intervals along the timeline, ignoring another important attribute of intervals: their duration.
%
As a result, index structures to support more general selection queries with constraints on both the duration and the position in time of intervals have been missing~\cite{DBLP:journals/sigmod/KulkarniM12,DBLP:journals/tods/DignosBGJ16,DBLP:conf/ebiss/BohlenDGJ17} until recently~\cite{BehrendDGSVRK_SSTD19_period-index}.	

This paper takes as its starting point the observation that in several application both aspects of temporal information are useful to formulate queries.
For instance, consider the following queries from different application scenarios:
\begin{itemize}
	\item[$Q2$:] \emph{"In a database of flights, find all the aircrafts that were flying between 10:00 and 14:00."}
  \item[$Q1$:] \emph{"In a source code repository, find all the files
    that have not been modified for more than two years."}
	\item[$Q3$:] \emph{"Find all the tourists who stayed in Bolzano between 2 and 4 weeks during August last year."}
\end{itemize}

The first query $Q1$ imposes constraints on the duration of matching events,
and we call it \emph{duration query}. In contrast, query $Q2$ retrieves
tuples based on the position of the events on the timeline; we call it
\emph{range query}. Finally, query $Q3$ constrains both types of information;
we call it \emph{range-duration query}.
Similar queries have been reported in
other application scenarios, e.g., air traffic
analysis~\cite{DBLP:conf/gis/SchullerBM10,DBLP:conf/adbis/BehrendMSW09,DBLP:conf/adbis/SchullerSB12}.

As shown in the above examples, time periods can comprise information about
their position on the timeline and the length of the period. However, index
structures over time intervals typically use only one of the two aspects. For
instance, the well-known relational interval
tree~\cite{DBLP:conf/vldb/KriegelPS00} is optimized for efficiently
determining temporal relationships between intervals but not interval
lengths. In the worst case, the entire index tree must be traversed if a
query solely contains restrictions on the interval length. 
Alternatively, the duration of the intervals can be indexed straighforwardly using a classic data structure such as a B-tree.
In this case, however, queries constraining only the range of the intervals will need to traverse the entire tree.

The two aforementioned indexes could be used at the same time, however combining different indexes is typically inefficient, since a query can have different selectivities in the two dimensions.
%
Therefore, to efficiently support queries constraining both the duration and the range of intervals we need to develop a data structure that can index both dimensions at the same time. A first step in this direction is \algo{Period-Index*}~\cite{BehrendDGSVRK_SSTD19_period-index}.

\jg{My feeling is that in this paragraph we need to provide a few more details, e.g., about our guarantees} We improve on this previous work by proposing \ours{}, a two
dimensional data structure that indexes both duration and position on
the timeline at the same time.  In particular, our data structure
partitions the intervals in a grid according to their start times and

which are proportional to the selectivity of the query.  The data
structure also lends itself to a rather simple implementation, which
we use to experimentally validate our approach using such
implementation, finding that the overhead introduced by the data
structure is indeed negligible, and that the running time in practice
is largely proportional to the selectivity of the query.  This is in
contrast to the other data structures we compare to.

Our contributions can then be summarized as follows:
\begin{itemize}
	\item We describe \ours{}, an index data structure that supports queries involving both the duration and range components of intervals;
	\item We prove bounds on its performance, which can be tuned with a single parameter;
	\item We provide an extensible open source implementation, which we benchmark against several baselines, showing significantly better performance across several workloads.
\end{itemize}

Very recently, Behrend et al.~\cite{BehrendDGSVRK_SSTD19_period-index}
proposed an index expliticly supporting range-duration queries. Such index,
named \algo{Period-Index*}, partitions the time domain into \emph{buckets},
assigning an interval to all the buckets it intersects. Within each bucket,
intervals are further partitioned in \emph{levels} according to their
duration, with the minimum duration indexed within each level decreasing
geometrically. To support efficient indexing of times, each level is further
partitioned in the time domain. This data structure is adaptive to the
distribution of start times, while it assumes a Zipf-like distribution for
the durations of the intervals. As we shall see, our index structure removes
this assumption, supporting databases with arbitrary distributions of
durations.

\section{Preliminaries}

We assume a linearly ordered, discrete time domain, $\domT$.  A time
interval is a set of contiguous time points, and $\T = [\TS,\TE)$
denotes the closed-open interval of points from $\TS$ to $\TE$.  We
use $|T| = \TE - \TS$ to denote the duration of time interval $\T$ and
$\T \cap \T'$ to denote the set of time points shared by two intervals
$T$ and $T'$, which, if not empty, is itself an interval.
%
The schema of a temporal relation is given by $R = (\args{A}{m},T)$,
where $\args{A}{m}$ are the non-temporal attributes with domains
$\Omega_i$ and $\T$ is the timestamp attribute with domain
$\domT \times \domT$ representing, for instance, the tuple's valid
time.  A temporal relation $\R$ with schema $R$ is a finite set of
tuples, where each tuple has a value in the appropriate domain for
each attribute in the schema. We use $r.A_i$ to denote the value of
attribute $A_i$ in tuple $r$.  We use the abbreviations
$\A = \{ \args{A}{m} \}$ and $r.\A = (\args{r.A}{m})$.


\subsection{Temporal Queries}

Our index efficiently supports the three following types of queries (defined as in~\cite{BehrendDGSVRK_SSTD19_period-index}).

\begin{definition}[Range query]
  Given a temporal interval $\T = [\TS, \TE)$ and a temporal relation
  $\R$, a \emph{range query} $Q(\R, \T)$ is defined as
  \[
    Q(\R, \T) = \left\{r \in \R : \rT \cap \T \ne \emptyset
    \right\}
  \]
\end{definition}

\begin{definition}[Duration query]
	Given a duration interval $\D = [\Dmin, \Dmax]$ and a temporal relation $\R$,
	a \emph{duration query} $Q(\R, \D)$ is defined as
	\[
		Q(\R, \D) = \left\{r \in \R : |\rT| \in [\Dmin, \Dmax] \right\}
	\]
\end{definition}

\begin{definition}[Range-duration query]
  Given a temporal interval $\T = [\TS, \TE)$, a duration interval
  $\D = [\Dmin, \Dmax]$, and a temporal relation $\R$, a
  \emph{range-duration query} $Q(\R, \T, \D)$ is defined as
  \[
    Q(\R, \T, \D) = \left\{r \in \R : \rT \cap \T \ne \emptyset \wedge
      |\rT| \in [\Dmin, \Dmax] \right\}
	\]
      \end{definition}

      A range query retrieves all tuples whose timestamp intersects
      with the query range.  A duration query retrieves all tuples
      whose timestamp has a duration that is between $d_{min}$ and
      $d_{max}$. A range-duration query is a combination of the former
      two.  In the remainder of the paper we will refer, for brevity,
      with $q = (\T, \D)$ to the pair of constraints used in
      range-duration queries, and in particular we will use $\qT$ to
      refer to the time range constraint, and $\qD$ to denote the
      duration constraint

\section{Data Structure}

\subsection{Overview}

An interval can be completely described by its duration and starting
time, therefore we can represent a set of temporal intervals as a set
of points in a two dimensional space, with one coordinate being the
start time and the other the duration.  A range-duration query $q$ in
such a space is the polygon containing all the solutions, which is
defined by all points $(t, d)$ (corresponding to the time interval
$[t, t + d)$) such that
\begin{align*}
  (q.\TS - d < t < q.\TE) \quad \land \quad (q.\Dmin \le d \le q.\Dmax)
\end{align*}

\begin{example}
  Figure~\ref{fig:example-plane} shows the tuples of
  Table~\ref{tab:example} in the two dimensional space, along other
  flights on the same day.  The red polygon represents the query of
  the previous example with query range $[\text{10:00},\text{14:00})$
  and duration $[\text{2h}, \text{4h}]$.  Note how the points (i.e.,
  the timestamp intervals of the tuples) are not uniformly
  distributed.
\end{example}

Our index is a grid, partitioning the tuples into disjoint buckets according
to the durations and the start times of their intervals.
The boundaries between cells are defined by using the empirical cumulative
distribution function of the duration and the starting times, so that
each cell contains the same number of intervals, which corresponds to the
\emph{page size} of our index.
This allows the data structure to adapt to the distribution of the input.

In the following we will focus on the timestamp attribute $\T$ of
relation $\R$: in the discussion we assume that each interval being
inserted in the index is associated with a reference to the tuple in
$R$ it belongs to.

\label{sec:index-construction}

\begin{algorithm}[t]
	\caption{\textsc{BuildIndex}$(I, s)$\label{alg:build-index}}
	\KwIn{A collection of intervals $I$; the page size parameter $s$}
	\KwOut{A grid $G$ partitioning intervals first by time, then by duration;
	a sorted collection of column boundaries $t_s^i$; a sorted collection of cell
	boundaries $d_{max}^{ij}$ for each column $i$}

	\BlankLine

	Sort $I$ by start start time\;
	\Let{$h$}{$0$}\;
	\Let{$i$}{$0$}\;
	\While{$h < |I|$}{
		\nllabel{line:column-break}
		\Let{$h'$}{\algo{NextBreakpoint}$(I, h, s^2, \T \rightarrow \TE)$}\;
		\Let{$\TE^{i}$}{$\displaystyle\argmax_{\T \in I[0..h']} \TE$} \algocomment{maximum end time in all the columns, including the current one}\;
		\Let{$\TS^{i}$}{$(I_{h'})_s$} \algocomment{maximum start time in column} \;
		\Let{$I'$}{$I_{h..h'}$}\;
		Sort $I'$ by duration\;

		\Let{$k$}{$0$}\;
		\Let{$j$}{$0$}\;
		\While{$k < |I'|$}{
			\nllabel{line:cell-break}
			\Let{$k'$}{\algo{NextBreakpoint}$(I', k, s, \T \rightarrow |\T|)$}\;
			\Let{$\Dmin^{ij}$}{$|I_{k}|$}\algocomment{minimum duration in the cell}\;
			\Let{$\Dmax^{ij}$}{$|I_{k'}|$}\algocomment{maximum duration in the cell}\;
			\Let{$G_{ij}$}{$I'_{k..k'}$, sorted by end time}\;
			\Let{$j$}{$j+1$}\;
			\Let{$k$}{$k'+1$}\;
		}
		\Let{$h$}{$h' + 1$}\;
		\Let{$i$}{$i + 1$}\;
	}

	\Return $G$\;
\end{algorithm}

\begin{algorithm}[t]
	\caption{\textsc{NextBreakpoint}$(I, h, b, k)$\label{alg:next-breakpoint}}
	\KwIn{A collection of intervals $I$, sorted according to the key function $k$; the current index $h$; a bucket size $b$; a \emph{key} function $k: I \to \domT$}
	\KwOut{The next split index in the collection $I$}

	\Let{$h'$}{$\min\{h + b, |I| - 1\}$}\;
	\uIf{$k(I_h) = k(I_{h'})$} {
		\nllabel{line:heavy-bucket}
		{\footnotesize look for the last interval with the same key:}\;
		\lWhile{$h' < |I| \wedge k(I_h) = k(I_{h'+1})$} {
			\Let{$h'$}{$h'+1$}
		}
	} \Else {
		\nllabel{line:split-bucket}
		{\footnotesize look backwards for the first interval with a different key:}\;
		\lWhile{$h' > h \wedge k(I_{h'}) = k(I_{h'+1})$} {
			\Let{$h'$}{$h'-1$}
		}
	}
	\Return $h'$\;
\end{algorithm}

Our grid structure partitions the dataset first along
either the start time or duration dimension, and then along the
other. The choice of which dimension to index first may of course
impact the performance, depending on the query workload.  To make the
exposition more concrete, in the following we will assume that the
start time dimension is partitioned first.  All the descriptions,
considerations, and proofs also hold with the dimensions swapped.

Let $s$ be the \emph{page size} parameter.
The goal of the following
construction is to ensure that each cell of the grid contains at most
$s$ intervals.
The purpose of this parameter is to allow the optimization of the data structure
to different scenario. In an external memory setting, the parameter $s$ might be
set to the disk block size. In a main memory scenario, it could be set
to that a cell fits in a cache line.
We divide the intervals into \emph{columns} based on their duration with two
goals: all the intervals with the same start time should be in the same
column, and each column should have at most $s^2$ intervals, unless
there are more than $s^2$ intervals with the same start time.  In the
latter case, all the intervals in the column shall have the same
start time. 
Partitioning start times in this way allows, intuitively, to then further partition
each column in $s$ cells of $s$ elements each\footnote{We could use two separate 
parameters to control the number of partitions in the two dimensions, but using $s$ and $s^2$ makes
for a simpler exposition, analysis and implementation.}.

The pseudocode to partition the intervals in this way is depicted in
Algorithm~\ref{alg:build-index}. First, we sort all the intervals by
increasing start time. Then, we proceed to defining the columns iteratively,
defining breakpoints in the collection of input intervals, starting at $0$.
Using Algorithm~\ref{alg:next-breakpoint} we define a range of intervals that
will be grouped in the same column. In particular, we look at the interval
$s^2$ positions away from the current one (line~\ref{line:column-break}). If
the two intervals have the same start time, then we scan forward until we
find the last interval with the same start time
(line~\ref{line:heavy-bucket}). Note that in this case all the intervals in
the column will have the same start time, and since it has more than $s^2$
intervals we deem it \emph{heavy}. Otherwise, if the two intervals have
different start times, we scan backwards until the breakpoint interval and
the immediately next one have different start times, to make sure that the
breakpoint does not split in two a contiguous run of equal start times
(line~\ref{line:split-bucket}). Note that in this case the column will
contain at most $s^2$ intervals. Once we found the boundaries of the current
column, its intervals are sorted by increasing duration. At this point, we
proceed similarly to find the breakpoints between cells of the column
(line~\ref{line:cell-break}): starting from the first interval of the column,
we look at the interval $s$ position away, scanning forward if we are looking
at a run of more than $s$ intervals with the same duration (thus defining a
\emph{heavy cell}), or backward in order not to split runs of same-duration
intervals. Once a cell is identifies this way, its intervals are sorted by
end time, and we proceed to the next cell. After processing an entire column,
we move to the next. Along with the grouping of intervals into columns and
cells, during the execution of the algorithm we collect some ancillary
information: the maximum start time and maximum end time of each column\mc{this is to fix}
(respectively denoted $\TS^i$ and $\TE^i$ for column $i$), and the maximum and minimum durations
in each cell $ij$, denote with $\Dmax^{ij}$ and $\Dmin^{ij}$.
This information will be used to efficiently index into the grid in order to 
answer range-duration queries.

\begin{example}
Figure~\ref{fig:example-partitioning} depicts a sequence of
invocations of Algorithm~\ref{alg:next-breakpoint} on a sequence
of sorted start times, with parameter $b=9$.
The first jump by 9 positions would split a run of intervals with the
same start time, therefore the algorithms iterates back until the first
start time $< 13$ (this corresponds to line~\ref{line:split-bucket} in Algorithm~\ref{alg:next-breakpoint}).
Then, the second invocation would again split a run of more than 9 intervals
with the same start time, but this time the endpoints of the jump have the
same value. We are therefore in the case handled by line~\ref{line:heavy-bucket}
in Algorithm~\ref{alg:next-breakpoint}, and thus we iterate forward
until the last interval with the same start time, defining a \emph{heavy}
column.
The last jump defines the third column
\end{example}

\subsection{Querying the Index}

Given a range-duration query $q$, recall that a tuple $r \in R$
satisfies the query if the three following conditions are met:
\begin{align}
	 & |\rT| \in [\qD_{min}, \qD_{max}] \label{eq:duration-condition} \\
	 & q.\TS < r.\TE                  \label{eq:start-condition}     \\
	 & r.\TS < q.\TE                  \label{eq:end-condition}
\end{align}

The query algorithm is reported in Algorithm~\ref{alg:query}.
First find the index of the last column in the start time dimension containing
at least one interval satisfying condition~\ref{eq:end-condition}, by means 
of binary search on the boundaries of the columns 
Let this index be $i$.
Then we inspect column $G_i$ and all the preceding columns, until we get to a column
that cannot possibly contain intervals matching the query, because the latest end time 
of that column \emph{and all the preceding ones}
does not satisfy condition~\ref{eq:start-condition}.
When inspecting a column, we look for the cell within the column with the largest 
minimum duration, again by binary search.
Then we scan the cells backward, util we get to a cell whose maximum duration is smaller than the
$\Dmin$ required by the query.
For each interval contained in the visited cells, we scan the intervals by decreasing end time,
until the end time no longer satisfies~\ref{eq:start-condition}, reporting
all the intervals satisfying the query.

\begin{algorithm}[t]
  \caption{\textsc{Query}$(q))$\label{alg:query}}
  \KwIn{A range duration query $q$; An index grid $G$}

  \BlankLine
  \Let{$i$}{$\argmax_{i}\TS^i < q.\TE$}\;
  \While{$i \ge 0 \wedge q.\TS < \TE^i$} {
	\Let{$j$}{$\argmax_{j}\Dmin^{ij} \le q.\Dmax$}\;
	\While{$\Dmax^{ij} \ge q.\Dmin$}{
		\For{$\T \in G_{ij}$}{
			\If{$\TE \le q.\TS$}{
				\textbf{break}\;
			}
			\If{$|\T| \in [\Dmin, \Dmax] \wedge \TS < q.\TE$}{
				\textbf{report} $\T$\;
			}
		}
		\Let{$j$}{$j-1$}\;
	}
	\Let{$i$}{$i-1$}\;
  }

\end{algorithm}


\subsection{Updating the index}
\label{sec:updating-index}

Our index data structure can be extended to support both insertion and
removals of tuples. For a tuple $r$, with $\rT=[\TS, \TE)$, by
querying the index for the duration $|\rT|$ and the start time $\TS$,
we can identify the cell that should contain the tuple $r$.  Inserting
new intervals directly into cells might make them grow too large to be
able to make performance guarantees on the query time.  Therefore, we
set a threshold $2s^2$ for \emph{light} columns and $2s$ for
\emph{light} cells (as we shall see in the next sections, we can bound
the performance of \emph{heavy} cells and columns even though they
exceed the page size).  If a light column exceeds size $2s^2$ then we
split the column, if a light cell exceeds size $2s$ then we split the
cell.

Splitting a column $G_i$ entails finding the median start time $\bar{\TS}$
among its intervals, and then replacing it
with two new columns with boundaries $(\TS^{i-1}, \bar{\TS}]$ and 
$(\bar{\TS},\TS^{i}]$, respectively.
Each of the two new columns requires the construction of new cell 
boundaries, which can be done by invoking
Algorithm~\ref{alg:next-breakpoint} repeatedly, like in the inner loop of
index construction.

Similarly, to split a cell we replace it with two new cells with boundaries
$(\Dmax^{i-1}, \bar{|\T|}]$ and $(\bar{|\T|}, \Dmax^{i})$, where $\bar{|\T|}$ is
the median duration in the cell being split. Then the intervals in each of
the two newly created cells are sorted by end time.

As for the removal of a tuple $r$ from the index, we query the index to find
the cell $G_ij$ containing $\rT$, removing the interval from the cell. As a
consequence the cell might be empty, in which case case we remove the cell
from the column. Finally, if the removal of the cell results in an empty
column, we remove the column.

\section{Analysis}

We now provide guarantees on the time required by all the operations
supported by our index structure. In this analysis we assume that the index
is built by partitioning first in the time dimension, and then in the
duration dimension. The choice of the order of this partitioning affects the
practical performance, as we will discuss in Section~\ref{sec:experiments}
(Figure~\ref{fig:param-dependency}). Nonetheless, the results in this
section hold for both dimension orderings.

\begin{theorem}
	Given a set of $n$ intervals, building the index requires time $\BO{n \log n}$.
\end{theorem}
\begin{proof}
  Computing the bucket boundaries for the duration using
  Algorithm~\ref{alg:define-boundaries} requires time $\BO{n\log n}$
  for sorting (line~\ref{line:sorting}) and time $\BO{n}$ for
  iterating through the sorted intervals to find the boundaries
  (line~\ref{line:iterate}): in fact, each of the $n$ intervals is
  visited at most once.

	Let $n_i$ be the number of intervals in the $i$-th column.
	Then, the invocation of Algorithm~\ref{alg:define-boundaries} to find cell boundaries
	requires time $\BO{n_i \log n_i}$, following the same argument as above.
	Therefore, the time for building the second level of the index is
	\[
          \BO{\sum_i n_i \log n_i}
          \le \BO{\log n \cdot \sum_i n_i}
          = \BO{n \log n}
        \]
	and the theorem follows.
\end{proof}

Before stating our main result, we establish the following fundamental facts
about heavy columns and cells.

\begin{lemma}\label{lem:heavy}
	A heavy cell in a heavy column contains only copies of the same interval.
\end{lemma}
\begin{proof}
	\mc{This is trivial, so we can omit it.}
	By construction, a heavy column contains only intervals with the same start time.
	Similarly, a heavy cell contains only intervals with the same duration.
	Therefore, a heavy cell in a heavy column 
	contains intervals with the same
	start time and duration, i.e. copies of the same interval.
\end{proof}

\begin{theorem}
	Given an index over a set of $n$ intervals, with page size $s$, the time for
	answering a range-duration query is
	\[
		\BO{\frac{n}{s^2}\log\frac{n}{s} + \frac{n}{s} + s^2 + k}
	\]
	where $k$ is the number of intervals matching the query predicate.
\end{theorem}
\begin{proof}
	Let $q.\T = [\TS, \TE)$ be the query interval, and $[\Dmin, \Dmax]$ be the duration range of the query.
	Recall that the index construction algorithm may create a heavy column or cell when it cannot break down a group of intervals
	because they all share the same start time or duration.
	A range-duration query defines a \emph{query polygon} in the start time $\times$ duration space, as depicted in Figure~\ref{fig:example-grid}.
	For a given cell $G_{ij}$, we denote by $k_{ij}$ the number of intervals in $G_{ij}$ matching the query.

	The query time is comprised of two parts: first we have to index into the grid to find the correct
	range of cells to query; second we have to filter the intervals in each cell to retrieve
	the ones that are part of the answer.

	As for finding the range of cells, we first have to find potential matches the last column by start time, which can be
	done by binary search over the $n/s^2$ boundaries between columns in $\BO{\log (n/s^2)}$ time.
	Iterating backwards through the columns visits at most $\BO{n / s^2}$ columns.
	In each such column, to find the last cell with intervals satisfying the duration constraint
	we do a binary search over the $\BO{n/s}$ cells in the column
	(since in the worst case we are inspecting a heavy column, which can have $\BO{n}$ intervals).
	Therefore, the overall time to find the correct cells is
	\[
		\BO{\log\frac{n}{s^2} + \frac{n}{s^2} \log\frac{n}{s}} =
		\BO{\frac{n}{s^2} \log\frac{n}{s}}
	\]

	Consider now the enumeration of the results from the cells. There are four possible cases for a cell $C$:
	\begin{enumerate}
		\item Both the cell's durations range and start times range fall
		within the query polygon. In such cell, the query constraints on the
		duration and the start time are satisfied by construction. The
		algorithm considers just the intervals of $G_{ij}$ which are part of
		the output (since it iterates over the cell's intervals by decreasing
		end time). Therefore, the time spent enumerating intervals from the
		cell is $\BO{k_{ij}}$.
		Also, note that this is the only case in which a heavy cell in a 
		heavy column is visited by the query algorithm. Such cell can potentially
		have $\BO{n}$ copies of the same interval. If the algorithm visits the cell, then
		it means that such copies are all part of the output, hence we pay 
		$\BT{k_{ij}}$ in the complexity.

		\item The cell's duration range satisfies the query constraint, but
		the starting time range doesn't. If the duration column containing the
		cell is heavy, then the cell cannot be heavy (by
		Lemma~\ref{lem:heavy}) and the algorithm has to evaluate $\BO{s}$
		intervals. Otherwise the cell might be heavy. In such case, all the
		start times are equal and satisfy the query constraint (otherwise we
		will not be looking at the cell at all). Therefore all the intervals
		in the cell match the query, and the time to enumerate them is
		$\BO{k_{ij}}$.

		\item The cell's start times all satisfy the query constraints, but
		not all its durations do. In this case we cannot be in a heavy column
		(which contains a single duration) and thus the column containing the
		cell contains at most $s^2$ intervals.

		\item The cell doesn't completely satisfy neither constraint. In this
		case neither the column nor the cell can be heavy, therefore the
		algorithm must only evaluate $\BO{s}$ intervals.
	\end{enumerate}
	All the cells of type (1) account for at most $\BO{k}$ in terms of running time.
	As for cells of type (2) there are at most $n/s^2$ of them. Overall they account for $\BO{\frac{n}{s}}$ interval evaluations.
	Cells of type (3) contain at most $s^2$ intervals. Therefore, overall they account for $\BO{s^2}$ interval evaluations.
	Finally there is a constant number of cells of type (4) (which are at the corners of the query polygon).
	These cells therefore account for at most $\BO{s}$ interval evaluations in total.
	Overall, the time to evaluate the intervals in the cells is
	\[
		\BO{\frac{n}{s} + s^2 + k}
	\]

	Combining this last result with the time to find the range of cells to query, we have that the overall query time is
	\[
		\BO{\frac{n}{s^2}\log\frac{n}{s} + \frac{n}{s} + s^2 + k}
	\]
\end{proof}

The above theorem exposes a fundamental tradeoff of our data structure:
using a smaller page size allows to improve the precision of the data structure
(by looking at fewer intervals that are not part of the query output), while
at the same time increasing the number of columns that need to be queried.

\begin{theorem}
	Inserting a tuple $r$ into the index requires
	time $\BO{\log n/s + n/s + s^2 \log s}$.
\end{theorem}
\begin{proof}
	Finding the cell that should contain $r.t$ requires time $\BO{\log\frac{n}{s^2}}$ to find the column, and
	time $\BO{\log\frac{n}{s}}$ to find the cell within the column (as in the worst case all the intervals
	in the index have the same duration).

  Note that a heavy column or cell does not need to be split, since by definition
  they can contain more than $s^2$ (resp. $s$) intervals, as long as they all have
  the same start time (resp. duration).
  Therefore we consider only the case of splitting a non-heavy column or non-heavy cell.
	Splitting a column with $2s^2 + 1$ intervals requires to find the median and
	to apply Algorithm~\ref{alg:define-boundaries} on the
	two newly defined columns, both operations that require time $\BO{s^2\log s^2}$.
	Furthermore, inserting the new column boundaries in order in the sorted array of boundaries requires
	$\BO{\frac{n}{s^2}}$ time.
	By similar reasoning, splitting a cell with $2s+1$ intervals requires time $\BO{s\log s}$,
	and updating cell boundaries by inserting in-order requires $\BO{\frac{n}{s}}$
\end{proof}

In a similar way we can prove the following theorem.

\begin{theorem}
	Removing a tuple $r$ from the index requires time $\BO{\log n/s + n/s}$
\end{theorem}

We remark that the overall difference in throughput for different workloads is
due to the different output sizes: \qdo queries are in general less selective than \qrd queries,
hence it takes more time to iterate through the output.
This explains why, in general, \qrd queries enjoy a higher throughput across all the index structures.

From the table we can see that \ours has in general a better performance compared to the other
structures.
In particular, it is up to one order of magnitude faster than the second-ranked baseline.

We also observe that the \algo{grid-file} ranks second in several cases.
Recall that this data structure is rather similar to \ours, the difference being
that the latter is adaptive to the input distribution.
This shows the performance benefits of a data structure that takes into account the data distribution.

On \qdo workloads, \algo{b-tree} is always the best performing index, up to one order of
magnitude faster than \ours.
However, on \qrd workloads, its performance is much worse.
As for the \algo{interval-tree} it shows performance comparable to the \algo{grid-file} on \qro
workloads, but is comparatively much slower on \qrd workloads.

We can therefore conclude that \ours is the index structure achieving the best compromise
for all three types of queries: \qrd, \qdo, and \qro.

Finally, we observe that the distribution of start times does not seem to play a crucial role
in determining the performance of the index structures we tested: similar workloads differing only by
their start time distribution show similar performance.

We investigate the relationship between the selectivity of the queries (i.e. the fraction of the
input that satisfies them) and the time taken by different data structures to answer them.
Given that range-duration queries constrain both the time and the duration ranges,
for a query $q$ we can define the \emph{time selectivity} as the fraction of the input that
satisfies the time range constraint of $q$, and the \emph{duration selectivity} as the fraction
of the input satisfying the duration constraint of $q$.

For a given dataset, we build a query set such that queries are uniformly distributed on the
\emph{time}$\times$\emph{duration} selectivity plan: this way we have queries which are
very selective in only one dimension, or that are very selective in both dimensions, or that are not
selective at all.
The goal is to investigate the behavior of each index structure for queries with different characteristics.
To account for the overhead of measuring time and to level out the effect of the CPU cache,
we measure the time taken to repeat a given query 100 consecutive times, which divided by 100
gives an estimate of the time required by the query.

Figure~\ref{fig:selectivity-grid} reports results for such a setup, with 1024 queries arranged in a 32$\times$32
grid, running on the \dataset{UZ Synthetic} dataset. Colors encode the time required by queries having
a particular combination of selectivity in the time and duration dimensions.
When reading Figure~\ref{fig:selectivity-grid}, remember that less selective queries require more time
just to iterate through the output.
Interestingly, different data structures exhibit different patterns in this plot, as a result
of how they access data.
The \algo{interval-tree} plot exhibits vertical bands: the data structure is only able to select intervals
based only on their position on the timeline, therefore for a fixed time selectivity of the query
the time employed to filter in the duration dimension does not depend on the selectivity of the query,
since all the candidate intervals need to be examined.
For similar reasons, \algo{b-tree} shows horizontal bands.
The data structures that explicitly index both dimensions, instead, tend to exhibit a more \emph{diagonal}
pattern, in particular \ours.
The pattern exhibited by \algo{period-index*} tends to be more similar to the \algo{interval-tree}.
This means that this index is more responsive to queries that are selective in the time dimension.
This is to be expected, since \algo{period-index*} is adaptive to the distribution of start times
in the dataset.
Its worse performance compared to \ours and \algo{grid-file} for queries of a given selectivity
is explained by the fact that some intervals might be represented multiple times in the index.

Figure~\ref{fig:time-vs-selectivity} reports the performance (in terms of query time) against
the \emph{overall} selectivity of the same queries.
Ideally, a data structure should answer the queries in time linearly dependent on the selectivity,
irrespective of whether the query is more selective in the time or duration dimension.
From the figure, we observe that \ours indeed shows the ideal behavior.
This is consistent with the theoretical analysis: the query time depends on the
output size plus some overhead proportional to the cell size, that can be amortized with
the correct parameter choice (which we will discuss in the next section).
Moreover, \ours is able to answer queries in a more timely fashion, compared to the other indices.

As for the \algo{grid-file}, the query times are more scattered: the efficiency of a query
in this case is in fact dependent on the density of the cells being considered.

Considering the \algo{b-tree} and \algo{interval-tree}, we have \qro and \qdo respectively
are answered with the worst performance, as we expect given that they are answered
by simple enumeration.
Other queries are answered with varying degrees of efficiency, with queries constraining
just the dimension on which the index specializes exhibiting a close-to-ideal behavior.

\subsection{Dependency on the page size parameter}
\label{sec:param-dep}

\begin{figure}
	\includegraphics[width=\columnwidth]{images/param_dependency.png}
	\caption{Dependency of the performance, in terms of queries per second, on the
		page size parameter.
		\label{fig:param-dependency}}
	\Description{}
\end{figure}

To test the dependency of the query time on the \emph{page size} parameter, we consider
a dataset with 10 million intervals with starting times both uniformly
distributed between 0 and 10 million 
and durations distributed according to a
Zipf's law with $\beta = 1$; and a dataset of the same size with 
Zipf-distributed start times and uniformly distributed durations.
The aim is to investigate the influence of the parameters when the skew is either in the
time or in the duration dimension.
As for the query workload, we consider batches of 20\,000 \qrd, \qro, and \qdo queries.

Figure~\ref{fig:param-dependency} reports the results of varying the order of the dimensions
index under these different workloads, in terms of queries per second.
We observe very different trends for different query workloads.
Consider first \qrd queries.
For data with uniform start times we have that there is an optimal value for the page size, which
is slightly smaller for the duration-time ordering.
For larger values the performance degrades.
If the start times are zipf-distributed, instead, the performance profile of the two orderings
is very different: indexing first the time and then the duration favors selecting large values
for the page size parameter, whereas indexing first the duration and then the start time shows
the opposite behavior.
A very similar trend can be observed for \qdo queries.
On the other hand, for \qro queries the index performs best when larger pages are used, under both
orderings.

The reason for this performance profile is to be found in the asymmetry of range-duration queries:
when querying for a time range, grouping the intervals by duration can help in examining fewer intervals,
since the overlap predicate can be rewritten as a function of the start time and the duration of the intervals;
conversely, the query for a duration range does not benefit from a partitioning of the time domain, since
the duration of an interval is independent of its position on the timeline.
Therefore, when evaluating \qro queries we benefit from a finer partitioning



If we move our attention, instead, to \qdo queries, we see
that increasing the page size is detrimental, with performance staying approximately the
same up until around page size 100.
This is also in line with our expectations: a larger page size means a coarser partitioning
of the dataset.
All the intervals in cells in the same duration column are considered as being part of the output,
therefore having less duration columns forces each query to evaluate more non-matching intervals,
thus increasing the running time.

Conversely, the performance improves with the page size for \qro queries.
In this case having fewer duration columns means that fewer groups of cells
will need to be queried, thus improving performance.

\section{Scalability with respect to the input size}
\label{sec:scalability}

\begin{figure*}
	\includegraphics[width=\textwidth]{images/scalability.png}
	\caption{Scalability of the index structures for the increasing dataset sizes.
		Synthetic datasets start from 10 million intervals up to one billion intervals.
		Each point corresponds to the best configuration of the given data structure for a particular dataset scale.
		The workload is of range-duration queries.
		\label{fig:scalability}}
	\Description{Scalability figure, showing that our proposed approach has superior performance for all tested
		dataset sizes, and scales gracefully.}
\end{figure*}

To test the scalability of the index structures, we consider datasets of increasing sizes.
For synthetic datasets we start with 10 million intervals, scaling by factors of 10 to one billion intervals.
For real-world datasets, we artificially increase their size as follows.
Given a dataset and a scale parameter $s$, let $\bar{t}$ the span of time covered by all the intervals in the dataset.
We make $s$ copies of each interval, and shift copy $i$ in time by $i\cdot \bar{t}$, for $i\in[0, s)$.
The underlying idea is to repeat the temporal patterns of the dataset on a longer time scale.

Figure~\ref{fig:scalability} reports the performance of the best configuration of each algorithm,
showing that the performance degrades gracefully as the dataset size increases, for all the data structures
involved in the comparison.
On synthetic datasets, for all tested sizes, \ours is about one order of magnitude faster than the
second-best data structure, \algo{grid-file}.

\begin{tcolorbox}
	\textbf{A note on my implementation of period-index-$\star$}.\\
	In~\cite{BehrendDGSVRK_SSTD19_period-index}, there is a note about how to save on
	empty cells, which are the vast majority in all the experiments we try (from 50\% up to 99\%).
	In the paper, there is the suggestion to use a linked structure, with non-empty buckets linked to one another.
	However, as I see it, this would imply that we lose the ability to index into the first non-empty bucket
	containing potentially overlapping intervals.
	Therefore, we would have to do a linear scan of these linked lists, which would absolutely kill performance.

	Given this, my implementation uses an array to hold all the cells, event the empty ones,
	thus allowing to directly index into it.
	Each cell, then holds a pointer to the next non-empty cell, allowing, at query time, to skip the empty cells.
	Still, we have problems of memory allocation, because we waste a great deal of space in empty cells.
\end{tcolorbox}
