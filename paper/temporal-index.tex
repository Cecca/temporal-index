\begin{abstract}
  Temporal information plays a crucial role in many database
  applications, however support for queries on such data is limited.
  We present an index structure, termed \rdindex, to support
  \emph{range-duration queries} over interval timestamped relations,
  which constrain both the \emph{range} of the tuple's start time
  along the timeline and its \emph{duration}.  \rdindex is based on a
  grid structure in the two-dimensional space, representing the
  timeline and the duration of timestamps, respectively.  It features
  provable bounds on the running time of all the operations, allows
  for a simple implementation, and sports very predictable query
  performance.  We benchmark our implementation on a variety of
  datasets and query workloads, investigating both the query rate and
  the behavior of the individual queries.  Our experimental evaluation
  shows that, compared to other data structures, our index indeed
  better supports range-duration queries.  While other data structures
  are faster on specialized workloads constraining only the duration
  or only the range, overall our data structure expresses the best
  compromise across all tested workloads.
\end{abstract}

\section{Introduction}

Temporal information plays a crucial role in many database applications: many database management systems provide indeed automated version control of the data and \emph{time travel} facilities, allowing to access past history efficiently.
Past research concentrated on the important operators of temporal aggregation~\cite{DBLP:conf/icde/KlineS95,DBLP:journals/tkde/MoonLI03,DBLP:conf/edbt/BohlenGJ06,DBLP:conf/ssd/PiatovH17}, temporal joins~\cite{DBLP:conf/icde/PiatovHD16}, and time travel~\cite{DBLP:conf/sigmod/KaufmannMVFKFM13}.

All these approaches focus on handling temporal intervals, but concentrate on the \emph{position} of intervals along the timeline, ignoring another important attribute of intervals: their duration.
%
As a result, index structures to support more general selection queries with constraints on both the duration and the position in time of intervals have been missing~\cite{DBLP:journals/sigmod/KulkarniM12,DBLP:journals/tods/DignosBGJ16,DBLP:conf/ebiss/BohlenDGJ17} until recently~\cite{BehrendDGSVRK_SSTD19_period-index}.	

In many application domains both aspects of temporal information are
useful to formulate queries.  For instance, consider the following
queries from the tourism sector:
\begin{itemize}
	\item[$Q1$:] \emph{"Find all the tourists who stayed in Bolzano the weekend of pentecost (June 8th--9th) 2019."}
	\item[$Q2$:] \emph{"Find all the tourists who stayed in Bolzano for a short term (between 1-3 days)."}
	\item[$Q3$:] \emph{"Find all the tourists who stayed in Bolzano between 2 and 4 weeks during August last year."}
\end{itemize}
%
% For instance, applications that offer
% situation awareness~\cite{DBLP:conf/iri/BehrendSGLF15} require fast access
% both to the duration and the position of events.
%
The first query $Q1$ retrieves tuples based on the position of the
events on the timeline; we call it \emph{range query}.  In contrast,
query $Q2$ imposes constraints on the duration of matching events, and
we call it \emph{duration query}.  Finally, query $Q3$ constrains both
types of information; we call it \emph{range-duration query}.  Similar
queries can easily be found and have been reported in other
application scenarios, such as source code repository analysis, or air
traffic
analysis~\cite{DBLP:conf/gis/SchullerBM10,DBLP:conf/adbis/BehrendMSW09,DBLP:conf/adbis/SchullerSB12}.

As shown in the above examples, time periods can be constrained in
their starting position on the timeline and the length of the
period. However, index structures over time intervals typically use
only one of the two aspects. For instance, the well-known relational
interval tree~\cite{DBLP:conf/vldb/KriegelPS00} is optimized for
efficiently determining temporal relationships between intervals but
not interval lengths. In the worst case, the entire index tree must be
traversed if a query solely contains restrictions on the interval
length.  Alternatively, the duration of the intervals can be indexed
straighforwardly using a classic data structure such as a B-tree.  In
this case, however, queries constraining only the range of the
intervals will need to traverse the entire tree.

The two aforementioned indexes could be used at the same time, however combining different indexes is typically inefficient, since a query can have different selectivities in the two dimensions.
%
Therefore, to efficiently support queries constraining both the duration and the range of intervals we need to develop a data structure that can index both dimensions at the same time. A first step in this direction is \algo{Period-Index*}~\cite{BehrendDGSVRK_SSTD19_period-index}.

\jg{My feeling is that in this paragraph we need to provide a few more details, e.g., about our guarantees} We improve on this previous work by proposing \ours{}, a two
dimensional data structure that indexes both duration and position on
the timeline at the same time.  In particular, our data structure
partitions the intervals in a grid according to their start times and
durations. Rather than being fixed, the boundaries between the cells
of the grid are positioned by taking into account the data

the following).  This allows obtain very predictable query times,
which are proportional to the selectivity of the query.  The data
structure also lends itself to a rather simple implementation, which
we use to experimentally validate our approach using such
implementation, finding that the overhead introduced by the data
structure is indeed negligible, and that the running time in practice
is largely proportional to the selectivity of the query.  This is in
contrast to the other data structures we compare to.

Our contributions can then be summarized as follows:
\begin{itemize}
	\item We describe \ours{}, an index data structure that supports queries involving both the duration and range components of intervals;
	\item We prove bounds on its performance, which can be tuned with a single parameter;
	\item We provide an extensible open source implementation, which we benchmark against several baselines, showing significantly better performance across several workloads.
\end{itemize}

\section{Related work}

The type of selection queries we are studying in this paper are queries with a conjunctive predicate, where one predicate in the conjunction restricts the position of intervals on the time line and the second the duration of intervals. In this section, we review indexing structures that are (partially) suitable for such selection queries, and also review structures similar to our approach that are used for interval joins.

Edelsbrunner's Interval Tree~\cite{edelsbrunner80} is one of the most popular indexing structure for intervals, which is asymptotically optimal for selection queries involving the overlap predicate, and for which also an implementation using standard relational database technology using B-tree indexes~\cite{DBLP:conf/vldb/KriegelPS00} exists. The shortcoming of this index is that, in contrast to our indexing structure, does not provide a mechanism to restrict the duration of intervals and thus can only solve one part of a range-duration query. A similar indexing structure is the segment tree~\cite{deBerg00}, which builds disjoint segments over intervals at the
leaf level using all start and end points in a relation, and recursively merges segments in intermediate nodes on the tree. This data structure originally designed for point or time travel queries over intervals, also supports selection queries with the overlap predicate, but similar to the interval tree does not allow to restrict the duration of intervals at the same time.

Very recently, Behrend et al.~\cite{BehrendDGSVRK_SSTD19_period-index}
proposed an index explicitly supporting range-duration queries. Such index,
named \algo{Period-Index*}, partitions the time domain into \emph{buckets},
assigning an interval to all the buckets it intersects. Within each bucket,
intervals are further partitioned in \emph{levels} according to their
duration, with the minimum duration indexed within each level decreasing
geometrically. To support efficient indexing of times, each level is further
partitioned in the time domain. This data structure is adaptive to the
distribution of start times, while it assumes a Zipf-like distribution for
the durations of the intervals. As we shall see, our index structure removes
this assumption, supporting databases with arbitrary distributions of
durations.

In recent years, algorithms for interval joins have been actively studied. Moreover, the duration of intervals is not considered in this works.

\section{Preliminaries}

We assume a linearly ordered, discrete time domain, $\domT$.  A time
interval is a set of contiguous time points, and $\T = [\TS,\TE)$
denotes the closed-open interval of points from $\TS$ to $\TE$.  We
use $|t| = \TE - \TS$ to denote the duration of time interval $\T$ and
$\T \cap \T'$ to denote the set of time points shared by two intervals
$\T$ and $\T'$, which, if not empty, is itself an interval.
%
The schema of a temporal relation is given by $R = (\args{A}{m},T)$,
where $\args{A}{m}$ are the non-temporal attributes with domains
$\Omega_i$ and $T$ is the timestamp attribute with domain
$\domT \times \domT$ representing, for instance, the tuple's valid
time.  A temporal relation $\R$ with schema $R$ is a finite set of
tuples, where each tuple has a value in the appropriate domain for
each attribute in the schema. We use $r.A_i$ to denote the value of
attribute $A_i$ in tuple $r$, and $\rT = [\rTS, \rTE)$ to refer to its
time interval.  We use the abbreviations $\A = \{ \args{A}{m} \}$ and
$r.\A = (\args{r.A}{m})$. \jg{Not sure whether these 2 abbreviations

\subsection{Temporal Queries}

Our index efficiently supports the three following types of queries (defined as in~\cite{BehrendDGSVRK_SSTD19_period-index}).

\begin{definition}[Range query]
  Given a temporal interval $\T = [\TS, \TE)$ and a temporal relation
  $\R$, a \emph{range query} $Q(\R, \T)$ is defined as
  \[
    Q(\R, \T) = \left\{r \in \R : \rT \cap \T \ne \emptyset
    \right\}
  \]
\end{definition}

\begin{definition}[Duration query]
	Given a duration interval $\D = [\dmin, \dmax]$ and a temporal relation $\R$,
	a \emph{duration query} $Q(\R, \D)$ is defined as
	\[
		Q(\R, \D) = \left\{r \in \R : |\rT| \in [\dmin, \dmax] \right\}
	\]
\end{definition}

\begin{definition}[Range-duration query]
  Given a temporal interval $\T = [\TS, \TE)$, a duration interval
  $\D = [\dmin, \dmax]$, and a temporal relation $\R$, a
  \emph{range-duration query} $Q(\R, \T, \D)$ is defined as
  \[
    Q(\R, \T, \D) = \left\{r \in \R : \rT \cap \T \ne \emptyset \wedge
      |\rT| \in [\dmin, \dmax] \right\}
	\]
\end{definition}

A range query retrieves all tuples whose timestamp intersects with the
query range $\T$.  A duration query retrieves all tuples whose
timestamp has a duration that is between $d_{min}$ and $d_{max}$. A
range-duration query is a combination of the former two.

\subsection{Overview}

Our index is a two-dimenstional grid, partitioning the tuples into
disjoint buckets according to the start time and the duration of the
intervals.  The boundaries between cells are defined by using the
empirical cumulative distribution function of the tuples' duration and
the starting times, so that each cell contains approximately the same
number of intervals, which corresponds to the \emph{page size} $s$ of
our index.  This allows the index to adapt to the distribution of the
input and to different scenarios.  In an external memory setting, the
parameter $s$ might be set to the disk block size. In a main memory
scenario, it could be set such that a cell fits in a cache line.

In the following we will focus on the timestamp attribute $\T$ of
relation $\R$: in the discussion we assume that each interval being
inserted in the index is associated with a reference to the tuple in
$\R$ it belongs to.

Our grid structure partitions an array of tuples first along either the start
time or duration dimensions of the intervals, and then along the other. 
The choice of
which dimension to index first may impact the performance of the
index, depending on the query workload and the data distribution, as we shall
see in Section~\ref{sec:experiments}.  In
the following we assume that the start time dimension is partitioned
first, followed by the duration dimension.  All the descriptions,
considerations, and proofs also hold with the dimensions swapped.

Before describing the algorithm to build the index, we describe
\textsc{NextSubseq}, a subroutine that partitions an array of tuples sorted
according to a key function. We will use this subroutine to define columns
and cells of our grid structure. In particular, given an index $h$ and a size
parameter $b$, \textsc{NextSubseq} returns a subsequence starting at $h$ that
either contains at most $b$ tuples or contains tuples all sharing the same
key. Additionally, all the tuples with the same key are part of the same
subsequence. The pseudocode is reported in
Algorithm~\ref{alg:next-breakpoint} and works as follows. Starting from index
$h$ we look at the tuple at position $h' = h + b$ and consider two cases. If
the tuples at position $h$ and $h'$ have the same key, then we scan forward
until the first tuple with a different key (line~\ref{line:heavy-bucket}).
Otherwise, we scan backward until
the tuples at $h'$ and $h'-1$ have different keys (line~\ref{line:split-bucket}).
In both cases, the
rationale is to avoid splitting runs of same-key tuples between different
subsequences.

\begin{example}
Figure~\ref{fig:example-partitioning} depicts a sequence of
invocations of Algorithm~\ref{alg:next-breakpoint} on a sequence
of sorted start times, with parameter $b=9$.
The first jump by 9 positions would split a run of intervals with the
same start time, therefore the algorithms iterates back until the first
start time $< 13$ (this corresponds to line~\ref{line:split-bucket} in Algorithm~\ref{alg:next-breakpoint}).
Then, the second invocation would again split a run of more than 9 intervals
with the same start time, but this time the endpoints of the jump have the
same value. We are therefore in the case handled by line~\ref{line:heavy-bucket}
in Algorithm~\ref{alg:next-breakpoint}, and thus we iterate forward
until the last interval with the same start time, defining a \emph{heavy}
column.
The last jump defines the third column
\end{example}


We are now ready to describe the index construction procedure, whose pseudocode 
is reported in Algorithm~\ref{alg:build-index}. Let $s$ be the
\emph{page size} parameter, and \Tuples be the array of all tuples to be
indexed. First, we sort \Tuples by increasing start time. Then, we divide the
intervals into \emph{columns} of $s^2$ tuples based on their start times by
repeatedly invoking \textsc{NextSubseq} (\ref{line:column-break}). Defining
columns in this way allows, intuitively, to then further partition each
column in $s$ cells of $s$ elements each\footnote{We could use two separate
parameters to control the number of partitions in the two dimensions, but
using $s$ and $s^2$ makes for a simpler exposition, analysis and
implementation.}. As a special case, if there are more than $s^2$ tuples with
the same start time, \textsc{NextSubseq} will assign them to the same column,
which we then deem \emph{heavy}.
We also keep track of the minimum start time in each column using
an auxiliary array \Colbounds, together with the cumulative maximum end 
time in the columns in array \LatestEnds.
Both this arrays will be used at query time: the first to find the first column
to inspect for a given query, the second to determine when to stop iterating
through columns.

Each column is further partitioned in \emph{cells} in a similar way. First we
sort the tuples in the column by increasing duration. Then we define cells of
size $s$ by repeatedly invoking \textsc{NextSubseq} (\ref{line:cell-break}).
Again, if there are more than $s$ tuples with the same duration, they will
all be assigned to the same cell, which will then be deemed \emph{heavy}.
Similarly to columns, also cells are complemented by two arrays of ancillary
information: \Cellbounds stores the minimum duration in each cell, to be used
at query time to find the first cell to inspect, while \MaxDurations stores
the maximum duration in each cell, which at query time will determine when to
stop iterating through cells.

Finally, tuples in each cell are sorted by the end time.

\subsection{Querying the Index}

Given a range-duration query with time range $[\TS, \TE)$, 
recall that a tuple $r \in \R$
satisfies the query if the three following conditions are met:
\begin{align}
	 & |\rT| \in [\dmin, \dmax] \label{eq:duration-condition} \\
	 & \TS < \rTE                  \label{eq:start-condition}     \\
	 & \rTS < \TE                  \label{eq:end-condition}
\end{align}

The query pseudocode is reported in Algorithm~\ref{alg:query}.
First, we find the index of the last column that can contain matching
tuples.
To this end, we perform a binary search on the array \Colbounds to find
the last column $i$ such that the minimum start time in the column (which is the column bound) is strictly less than \TE, the query end time (line~\ref{line:search-column}).
This ensures that the column contains at least one tuple satisfying
condition \eqref{eq:end-condition}.
Then, we iterate \emph{backwards} through columns until column $i$
cannot possibly contain tuples satisfying the query.
For this, we use the support array \LatestEnds: if
$\LatestEndsi \le \TS$ then we know that all the columns at index $\le i$
contain tuples stopping earlier than the start of the query range,
hence we can avoid inspecting them because of condition~\eqref{eq:start-condition}.

For each column that we consider, we perform a binary search on the array
\Cellboundsi, looking for the last cell $j$ such that the minimum duration in
the cell (which is the cell bound) is $\le$ to $\dmax$, the maximum duration
in the query (line~\ref{line:search-cell}). Doing so ensures that at least
one tuple in the cell satisfies the upper bound of condition
\eqref{eq:duration-condition}. Then, we iterate backwards through the cells,
until we reach a cell whose maximum duration is less than the minimum
duration of the query. At this point we stop, since
condition~\eqref{eq:duration-condition} can no longer be satisfied.

Finally, we iterate through tuples of each considered cell by decreasing
end time, stopping as soon as condition~\ref{eq:start-condition} is no longer
satisfied (line~\ref{line:inner-break}), reporting all the intervals
satisfying the query.

\begin{algorithm}[t]
  \caption{\textsc{Query}\label{alg:query}} \KwIn{A range duration query with time range $[\TS, \TE)$ and duration range $[\dmin, \dmax]$; An index \Grid with the ancillary arrays \Colbounds, \Cellbounds, \LatestEnds, and \MaxDurations}

  \BlankLine
	\nllabel{line:search-column}
  \Let{$i$}{$\argmax_{i} \Colboundsi < \TE$}\;
  \While{$i \ge 0 \wedge \TS < \LatestEndsi$} {
	\nllabel{line:search-cell}
	\Let{$j$}{$\argmax_{j} \Cellboundsij \le \dmax$}\;
	\While{$j \ge 0 \wedge \MaxDurationsij \ge \dmin$}{
		\For{$r \in \Gridij$}{
			\nllabel{line:inner-break}
			\If{$\rTE \le \TS$}{
				\textbf{break}\;
			}
			\If{$|\rT| \in [\dmin, \dmax] \wedge \rTS < \TE$}{
				\textbf{report} $r$\;
			}
		}
		\Let{$j$}{$j-1$}\;
	}
	\Let{$i$}{$i-1$}\;
  }

\end{algorithm}


\subsection{Updating the index}
\label{sec:updating-index}

Our index data structure can be extended to support both insertion and
removals of tuples. For a tuple $r$, we query the index for the start time
\rTS and the duration $|\rT|$, identifying the cell that should contain $r$.
Inserting
new intervals directly into cells might make them grow too large to be
able to make performance guarantees on the query time.  Therefore, we
set a threshold $2s^2$ for \emph{light} columns and $2s$ for
\emph{light} cells (as we shall see in the next sections, we can bound
the performance of \emph{heavy} cells and columns even though they
exceed the page size).  If a light column exceeds size $2s^2$ then we
split the column, if a light cell exceeds size $2s$ then we split the
cell.

Splitting a column entails first finding the median start time $\bar{\TS}$
among its intervals.
Then we define two new columns, containing all the intervals with start time
strictly less than $\bar{\TS}$ and $\ge \bar{\TS}$, respectively.
The array \Colbounds is updated by inserting $\bar{\TS}$ at the appropriate
position, and the array \LatestEnds is updated accordingly.
Each of the two new columns requires the definition of new cells
which can be done by invoking
Algorithm~\ref{alg:next-breakpoint} repeatedly, like in the inner loop of
index construction.

Similarly, to split a cell we find its median duration $\bar{|\T|}$ and define
two new cells, updating \Cellboundsi (for a cell in the $i$-th column)
so that it contains $\bar{|\T|}$ as a boundary point.
\MaxDurations is updated similarly.
Then the intervals in each of
the two newly created cells are sorted by end time.

As for the removal of a tuple $r$ from the index, we query the index to find
the cell $\Gridij$ containing $\rT$, removing the interval from the cell. As a
consequence the cell might be empty, in which case case we remove the cell
from the column. Finally, if the removal of the cell results in an empty
column, we remove the column.

\section{Analysis}

We now provide guarantees on the time required by all the operations
supported by our index structure. In this analysis we assume that the index
is built by partitioning first in the time dimension, and then in the
duration dimension.
The asymptotic results in this
section hold for both dimension orderings.

\begin{theorem}
	Given a set of $n$ intervals, building the index requires time $\BO{n \log n}$.
\end{theorem}
\begin{proof}
  Computing column boundaries for the duration requires
  time $\BO{n\log n}$ for sorting the tuples, while invoking
  Algorithm~\ref{alg:next-breakpoint} requires time 
  $\BO{n}$ for iterating through the sorted intervals to find the boundaries: in fact, each of the $n$ intervals is
  visited at most once.

	Let $n_i$ be the number of intervals in the $i$-th column.
	Sorting the column requires time $\BO{n_i\log n_i}$
	Then, the invocation of Algorithm~\ref{alg:next-breakpoint} to find cell boundaries
	requires time $\BO{n_i}$, following the same argument as above.
	Similarly, let $n_{ij}$ be the size of cell $ij$. Sorting its tuples by end time requires time $\BO{n_{ij} \log n_{ij}}$.
	Therefore, the time for running the inner loop is
	$\BO{n \log n}$ overall,
	and the theorem follows.
\end{proof}

Before stating our main result, we establish the following fundamental facts
about heavy columns and cells.

\begin{lemma}\label{lem:heavy}
	A heavy cell in a heavy column contains only copies of the same interval.
\end{lemma}
\begin{proof}
	By construction, a heavy column contains only intervals with the same start time.
	Similarly, a heavy cell contains only intervals with the same duration.
	Therefore, a heavy cell in a heavy column 
	contains intervals with the same
	start time and duration, i.e. copies of the same interval.
\end{proof}

\begin{theorem}
	Given an index over a set of $n$ intervals, with page size $s$, the time for
	answering a range-duration query is
	\[
		\BO{\frac{n}{s^2}\log\frac{n}{s} + \frac{n}{s} + s^2 + k}
	\]
	where $k$ is the number of intervals matching the query predicate.
\end{theorem}
\begin{proof}
	Let $\T = [\TS, \TE)$ be the query interval, and $[\dmin, \dmax]$ be the duration range of the query.
	Recall that the index construction algorithm may create a heavy column or cell when it cannot break down a group of intervals
	because they all share the same start time or duration.
	A range-duration query defines a \emph{query polygon} in the start time $\times$ duration space, as depicted in Figure~\ref{fig:example-grid}.
	For a given cell $\Gridij$, we denote by $k_{ij}$ the number of intervals in $\Gridij$ matching the query.

	The query time is comprised of two parts: first we have to index \ad{index?} into the grid to find the correct
	range of cells to query; second we have to filter the intervals in each cell to retrieve
	the ones that are part of the answer.

	As for finding the range of cells, we first have to find potential matches the last column by start time\ad{I don't understand thus previous sentence}, which can be
	done by binary search over the $n/s^2$ boundaries between columns in $\BO{\log (n/s^2)}$ time.
	Iterating backwards through the columns visits at most $\BO{n / s^2}$ columns.
	In each such column, to find the last cell with intervals satisfying the duration constraint
	we do a binary search over the $\BO{n/s}$ cells in the column
	(since in the worst case we are inspecting a heavy column, which can have $\BO{n}$ intervals).
	Therefore, the overall time to find the correct cells is
	\[
		\BO{\log\frac{n}{s^2} + \frac{n}{s^2} \log\frac{n}{s}} =
		\BO{\frac{n}{s^2} \log\frac{n}{s}}
	\]

	Consider now the enumeration of the results from the cells. There are four possible cases for a cell $\Gridij$:
	\begin{enumerate}
		\item Both the cell's durations range and start times range fall
		within the query polygon. In such cell, the query constraints on the
		duration and the start time are satisfied by construction. The
		algorithm considers just the intervals of $\Gridij$ which are part of
		the output (since it iterates over the cell's intervals by decreasing
		end time). Therefore, the time spent enumerating intervals from the
		cell is $\BO{k_{ij}}$.
		Also, note that this is the only case in which a heavy cell in a 
		heavy column is visited by the query algorithm. Such cell can potentially
		have $\BO{n}$ copies of the same interval. If the algorithm visits the cell, then
		it means that such copies are all part of the output, hence we pay 
		$\BT{k_{ij}}$ in the complexity.

		\item The cell's duration range satisfies the query constraint, but
		the starting time range doesn't. If the duration column containing the
		cell is heavy, then the cell cannot be heavy (by
		Lemma~\ref{lem:heavy}) and the algorithm has to evaluate $\BO{s}$
		intervals. Otherwise the cell might be heavy. In such case, all the
		start times are equal and satisfy the query constraint (otherwise we
		will not be looking at the cell at all). Therefore all the intervals
		in the cell match the query, and the time to enumerate them is
		$\BO{k_{ij}}$.

		\item The cell's start times all satisfy the query constraints, but
		not all its durations do. In this case we cannot be in a heavy column
		(which contains a single duration) and thus the column containing the
		cell contains at most $s^2$ intervals.

		\item The cell doesn't completely satisfy neither constraint. In this
		case neither the column nor the cell can be heavy, therefore the
		algorithm must only evaluate $\BO{s}$ intervals.
	\end{enumerate}
	All the cells of type (1) account for at most $\BO{k}$ in terms of running time.
	As for cells of type (2) there are at most $n/s^2$ of them. Overall they account for $\BO{\frac{n}{s}}$ interval evaluations.
	Cells of type (3) contain at most $s^2$ intervals. Therefore, overall they account for $\BO{s^2}$ interval evaluations.
	Finally there is a constant number of cells of type (4) (which are at the corners of the query polygon).
	These cells therefore account for at most $\BO{s}$ interval evaluations in total.
	Overall, the time to evaluate the intervals in the cells is
	\[
		\BO{\frac{n}{s} + s^2 + k}
	\]

	Combining this last result with the time to find the range of cells to query, we have that the overall query time is
	\[
		\BO{\frac{n}{s^2}\log\frac{n}{s} + \frac{n}{s} + s^2 + k}
	\]
\end{proof}

The above theorem exposes a fundamental tradeoff of our data structure:
using a smaller page size allows to improve the precision of the data structure
(by looking at fewer intervals that are not part of the query output), while
at the same time increasing the number of columns that need to be queried.

The choice of the order of this partitioning affects the
practical performance, as we will also discuss in Section~\ref{sec:experiments}
(Figure~\ref{fig:param-dependency}). 

\begin{observation}\label{obs:dimension-order}
While changing the order in which dimensions are indexed does not change the
asymptotic behavior of \ours, the choice of such order affects the practical performance.
In fact range and duration
constraints play different roles in queries. When querying for durations, the
position of start times on the timeline does not affect the outcome of a
query. Conversely, a range query involves comparing both the start and the
end time of intervals: such comparison can be rephrased in terms of start
times and durations. Therefore, range queries can benefit from a partition of
the duration dimension, since it implies a partition of the end times.
Furthermore, duration constraints are typically much more selective than
range constraints. Therefore, indexing first by duration might imply that a
query has to iterate through fewer columns, compared to the case where start
times are indexed first.
\end{observation}


\begin{theorem}
	Inserting a tuple $r$ into the index requires
	time $\BO{\log n/s + n/s + s^2 \log s}$.
\end{theorem}
\begin{proof}
	Finding the cell that should contain $\rT$ requires time $\BO{\log\frac{n}{s^2}}$ to find the column, and
	time $\BO{\log\frac{n}{s}}$ to find the cell within the column (as in the worst case all the intervals
	in the index have the same duration).

  Note that a heavy column or cell does not need to be split, since by definition
  they can contain more than $s^2$ (resp. $s$) intervals, as long as they all have
  the same start time (resp. duration).
  Therefore we consider only the case of splitting a non-heavy column or non-heavy cell.
	Splitting a column with $2s^2 + 1$ intervals requires to find the median,
	to sort the newly created columns, and
	to apply Algorithm~\ref{alg:next-breakpoint} on them
	operations which require time $\BO{s^2\log s^2}$ overall.
	Furthermore, inserting the new column boundaries in order in the sorted array of boundaries requires
	$\BO{\frac{n}{s^2}}$ time.
	By similar reasoning, splitting a cell with $2s+1$ intervals requires time $\BO{s\log s}$,
	and updating cell boundaries by inserting in-order requires $\BO{\frac{n}{s}}$
\end{proof}

In a similar way we can prove the following theorem.

\begin{theorem}
	Removing a tuple $r$ from the index requires time $\BO{\log n/s + n/s}$
\end{theorem}

\section{Experimental evaluation}
\label{sec:experiments}

\newcommand{\btree}{\algo{B-Tree}\xspace}
\newcommand{\itree}{\algo{Interval-Tree}\xspace}
\newcommand{\gfile}{\algo{Grid-File}\xspace}
\newcommand{\pindex}{\algo{Period-Index$\star$}\xspace}
\newcommand{\rdtd}{\algo{RD-index-td}\xspace}
\newcommand{\rddt}{\algo{RD-index-dt}\xspace}

\begin{figure*}[t]
	\includegraphics[width=\textwidth]{images/real-distributions.png}
	\caption{Histograms of the start times (top) and durations (bottom) of the three real-world
		datasets considered in this paper.\label{fig:data-distributions}}
\end{figure*}

In this section we evaluate experimentally our proposed index.
To frame this evaluation, we consider the following data structures as baselines.
The implementation of \btree provided by the Rust standard library, which is optimized for CPU cache usage. Intervals are indexed by duration in this case.
The \itree index~\cite{DBLP:conf/vldb/KriegelPS00}, which we implemented ourselves, indexing intervals by their position on the timeline.
The \gfile~\cite{DBLP:journals/tods/NievergeltHS84} and \pindex{}~\cite{BehrendDGSVRK_SSTD19_period-index}, which we also implemented, and which index both start times and durations.
Our proposed data structure will be denoted with \rdtd when start times are indexed before end times, and with \rddt otherwise. 
In cases where the order of dimensions is not relevant to the discussion, we will use \ours instead.
To account for the potential shortcomings of our implementations, we will also evaluate the relative performance of the data structures with implementation-independent metrics~\cite{DBLP:journals/kais/KriegelSZ17}.

We aim to answer the following questions:
\begin{enumerate}
	\item How robust are the indices over different workloads? (\textsection~\ref{sec:overview})
	\item How do query times relate with the selectivity of the query? (\textsection~\ref{sec:query-inspect})
	\item How does the page size parameter affect the performance of the index? (\textsection~\ref{sec:param-dep})
	\item How does the performance scale with respect to the size of the dataset? (\textsection~\ref{sec:scalability})
\end{enumerate}

\subsection{Setup and datasets}

We implemented our index, as well as baseline competitors,
using Rust 1.44.1. Our implementation, which is open source and available
at~\url{https://github.com/Cecca/temporal-index}, allows to easily tune the parameters to explore
different questions~\cite{DBLP:conf/sisap/AumullerC20}.
The experiments presented in this section have been run on a machine
equipped with 94 Gb of memory and a Intel\textregistered Xeon\textregistered CPU E5-2667 v3 @ 3.20GHz processor.

As a benchmark we consider the following datasets and workloads\footnote{
	All the datasets are available in the code repository.}:
%
\paragraph{\dataset{Flight}} A set of 701\,353 flights, identified by their
takeoff and landing time, at the granularity of minute, covering the month
of August 2018.
Queries on this dataset are generated at random, with time ranges with durations
uniformly distributed between 1 and 31 days, at the granularity of hours,
and duration ranges with endpoints uniformly distributed between 1 minute and
a day, at the granularity of minutes.
%
\paragraph{\dataset{Webkit}} 1\,547\,419 file edits in the Webkit source code repository.
Intervals represent the timespan between successive edits to a file.
Queries on this dataset are generated at random, with time ranges with durations uniformly distributed between
one minute and one year,
and duration ranges with a maximum up to three years.
%
\paragraph{\dataset{Tourism}} 835\,072 stays in the city of Bolzano, from 2015 to 2020.
Each stay is represented by the day of arrival and the day of departure.
Queries, generated at random, have time ranges with durations uniformly distributed in $[1, 90]$ days,
and duration ranges have endpoints uniformly distributed in $[1, 120]$ days.
%
\paragraph{\dataset{Synthetic}} Randomly generated datasets.
Intervals start times are either uniformly distributed in $[1, n]$, where $n$ is the size of the dataset;
or are clustered around 10 instants sampled uniformly at random from $[1, n]$, following a $N(\mu_i, 10^7)$
distribution for $1 \le i \le 10$.
In both cases, interval durations follow a Zipf distribution with $\beta=1$.

Figure~\ref{fig:data-distributions} reports the distribution of start
times and durations of the three real-world datasets we consider in
this experimental evaluation. Note, in particular, that the start
times of the \dataset{Flight} dataset are approximately uniformly
distributed, with durations geometrically distributed.  The
\dataset{Tourism} dataset, on the other hand, has start times more
concentrated around the weekends, and duration values of three days,
one week and two weeks being the most frequent.  Finally, the
\dataset{Webkit} dataset has a distribution of durations similar to
\dataset{Flight}, but the distribution of start times is skewed
towards recent times, meaning that more file edits have been done
recently.

In particular, we run the following type of query:
\begin{lstlisting}[basicstyle=\ttfamily, mathescape=true]
SELECT COUNT(*) FROM r
 WHERE $\rTS < \TE$ AND $\TS < \rTE$
   AND $\dmin \le |\rT|$ AND $|\rT| \le \dmax$;
\end{lstlisting}
where $[\TS, \TE)$ is the range constraint and
$[\dmin, \dmax]$ is the duration constraint.
Evaluating a counting query, rather than reporting the matching tuples, allows to measure
the time to actually retrieve the tuples, rather than the time to print them to screen or
write them to a file.

\label{sec:overview}

In the first set of experiments, we consider both real-world and synthetic datasets
(with 10 million intervals).
Table~\ref{tbl:overview-qps} reports an overview
on the performance, in terms of queries per second, of
different index structures under the different combinations of dataset/query workload.
For index structures that take parameters, we report on the best configuration.
In particular, we defer the discussion of the effect of different
parameterizations of \ours to Section~\ref{sec:param-dep}.

We remark that the overall difference in throughput for different workloads is
due to the different output sizes: \qdo queries are in general less selective than \qrd queries,
hence it takes more time to iterate through the output.
This explains why, in general, \qrd queries enjoy a higher throughput across all the index structures.

From the table we can see that \ours has in general a better performance compared to the other
structures.
In particular, it is up to one order of magnitude faster than the second-ranked baseline.

We also observe that the \algo{grid-file} ranks second in several cases.
Recall that this data structure is rather similar to \ours, the difference being
that the latter is adaptive to the input distribution.
This shows the performance benefits of a data structure that takes into account the data distribution.

On \qdo workloads, \algo{b-tree} is always the best performing index, up to one order of
magnitude faster than \ours.
However, on \qrd workloads, its performance is much worse.
As for the \algo{interval-tree} it shows performance comparable to the \algo{grid-file} on \qro
workloads, but is comparatively much slower on \qrd workloads.

We can therefore conclude that \ours is the index structure achieving the best compromise
for all three types of queries: \qrd, \qdo, and \qro.

Finally, we observe that the distribution of start times does not seem to play a crucial role
in determining the performance of the index structures we tested: similar workloads differing only by
their start time distribution show similar performance.

\subsection{Distribution of query times against selectivity}
\label{sec:query-inspect}

		\label{fig:selectivity-grid}}
	\Description{Time versus selectivity in two dimensions}
\end{figure*}

\begin{figure*}
	\includegraphics[width=\textwidth]{images/selectivity-dep.png}
	\caption{
		% Note that for the period-index* duration only queries are the worst, because they need
		% to visit all the buckets. On top of that, most of the queries have to examine
		% each interval more than once.
		Number of examined intervals against selectivity of the query.
		Each point represents a query:
		red dots are \qro queries, blue dots are \qdo queries.
		The black line represents the ideal behavior, in which only matching

	\Description{Time versus selectivity}
\end{figure*}

We investigate the relationship between the selectivity of queries (i.e. the fraction of the
input that satisfies them) and the time taken by different data structures to answer them.
Given that range-duration queries constrain both the time and the duration ranges,
for a query we can define the \emph{time selectivity} as the fraction of the input that
satisfies the time range constraint of the query, and the \emph{duration selectivity} as the fraction
of the input satisfying the duration constraint of the query.

For a given dataset, we build a query set such that queries are uniformly distributed on the
\emph{time}$\times$\emph{duration} selectivity plane: this way we have queries which are
very selective in only one dimension, or that are very selective in both dimensions, or that are not
selective at all.
The goal is to investigate the behavior of each index structure for queries with different characteristics.
To account for the overhead of measuring time and to level out the effect of the CPU cache,
we measure the time taken to repeat a given query 100 consecutive times, which divided by 100
gives an estimate of the time required by the query.

Figure~\ref{fig:selectivity-grid} reports results for such a setup, with 1024 queries arranged in a 32$\times$32
grid, running on the \dataset{UZ Synthetic} dataset. In the top row of plots, colors encode the time required by queries having
a particular combination of selectivity in the time and duration dimensions.
In the bottom row of plots, colors encode the fraction of the dataset inspected by each query.
This metric allows a more implementation-independent assessment of the relative performance of different data structures.
When reading Figure~\ref{fig:selectivity-grid}, remember that less selective queries require more time
just to iterate through the output.
Interestingly, different data structures exhibit different patterns in this plot, as a result
of how they access data.
First and foremost, we note that each data structure has similar patterns
both in the top and the bottom plot, since the time to answer a query is
tightly related to fraction of intervals inspected.

The \algo{interval-tree} plot exhibits vertical bands: the data structure is able to select intervals
only based on their position on the timeline, therefore for a fixed time selectivity of the query
the fraction of intervals inspected (and thus the time to answer the query)
do not depend on the selectivity in the duration dimension,
since all the candidate intervals need to be examined.
For similar reasons, \algo{b-tree} shows horizontal bands.
Data structures that explicitly index both dimensions, instead, tend to exhibit a more \emph{diagonal}
pattern, in particular \ours.
The pattern exhibited by \algo{period-index*} tends to be more similar to the \algo{interval-tree}.
This means that this index is more responsive to queries that are selective in the time dimension.
This is to be expected, since \algo{period-index*} is adaptive to the distribution of start times
in the dataset.
Its worse performance compared to \ours and \algo{grid-file} for queries of a given selectivity
is explained by the fact that some intervals might be represented multiple times in the index.

Figure~\ref{fig:time-vs-selectivity} reports the performance against
the \emph{overall} selectivity of the same queries.
The performance is assessed in terms of the fraction of the input inspected
by each query, which are represented as dots whose position along the $x$
axis encodes their overall selectivity.
Ideally, a data structure should answer queries by inspecting just the
tuples which are parts of the output: such ideal behavior is represented
by the black diagonal in Figure~\ref{fig:time-vs-selectivity}.
From the figure, we observe that \ours indeed shows the ideal behavior,
when parameters are configured optimally.

As for the \algo{grid-file}, since the efficiency in answering a query depends on the density of the cells being considered, the performance is in many cases far from ideal, which can be seen from the fact that several queries are far away from the ideal diagonal.

Considering the \algo{b-tree} and \algo{interval-tree}, we have \qro and \qdo respectively
are answered with the worst performance (i.e. they score 1 in Figure~\ref{fig:time-vs-selectivity}), as we expect given that they are answered
by simple enumeration of the entire dataset.
Other queries are answered with varying degrees of efficiency, with queries constraining
just the dimension on which the index specializes (duration for \algo{b-tree} and time for \algo{interval-tree}) exhibiting a the ideal behavior.

Finally, \algo{period-index$\star$} inspects the same tuples multiple times for the majority of queries. This is because the index is storing several copies of each tuple.
As we observed previously, queries restricting only the time are the ones answered most efficiently
by this data structure.

\begin{wrapfigure}{r}{0.35\columnwidth}
	\centering
	\hspace{-3em}
	\includegraphics[width=0.4\columnwidth]{images/selectivity-dep-inefficient.png}
	\vspace{-1em}
\end{wrapfigure}
So far, we have reported on the performance of the best configuration
of \ours{}, observing that under both orderings of dimensions it visits
a fraction of intervals close to the selectivity of the query,
i.e. it shows close to optimal behavior.
The figure at the side reports, in the same scenario of
Figure~\ref{fig:time-vs-selectivity}, the performance of \ours{}
with a time-duration ordering and page size 10.
As we will see in the next section, this configuration is not ideal:
indeed, for several queries we have to visit more intervals than the
precision of the query.
This is especially true for duration-only queries (blue dots). 
Nevertheless, even under this non-optimal setting, \ours still performs better than the other approaches,
i.e. points are more concentrated towards the diagonal.

\subsection{Dependency on the page size parameter}
\label{sec:param-dep}

\begin{figure}
	\includegraphics[width=\columnwidth]{images/param_dependency.png}
	\caption{Dependency of the performance, in terms of queries per second, on the
		page size parameter.
		\mc{Try also skew-skew and uniform uniform}
		\label{fig:param-dependency}}
	\Description{}
\end{figure}

To test the dependency of the query time on the \emph{page size} parameter, 
we consider two datasets with 10 million intervals each. The first datasets has 
uniform start times and skewed durations (with Zipf distribution), the second
has skewed start times and uniformly distributed durations.
As for the query workload, we consider batches of 10\,000 \qrd, \qro, and \qdo queries.
The aim is to investigate the influence of the parameters in different scenarios, in particular the influence
of the order in which dimensions are indexed.
We vary the page size between 1 and 10\,000. Note that, due to the size of the dataset, any page size above 
$\sqrt{10\cdot 10^6} \approx 3\,162$ results in a degenerate configuration where
there is a single column containing all the intervals. In this case the intervals are 
partitioned according to a single dimension, with the number of cells controlled by the page size parameter.
This situation might happen also at lower values of the page size parameter, depending on the number 
of distinct values in the dimension being partitioned.
On the other hand, for page size $1$, each cell of the grid contains all and only intervals with the
same start time and duration. Querying the data structure in this case amounts to performing binary searches
directly on the values of the domains of start times and durations.

Figure~\ref{fig:param-dependency} reports the results of this experiment,
in terms of queries per second. To ease the comparison between the subplots, we rescale
the reported throughput by the highest value for a each combination of dataset and query workload.
Degenerate configurations resulting in a single column are reported as triangles rather than dots.

We observe very different trends for different query workloads.
Consider first \qrd queries on the dataset with skewed durations.
In this case, both orderings of dimensions exhibit a similar behavior, with peak performance
reached by intermediate values of the page size.
If we consider \qdo queries, instead, the profile changes. Indexing first by duration (blue line) slightly
favors smaller page sizes, which imply smaller columns and thus a more fine grained access to the data.
Indexing first by time, instead, requires a \qdo to traverse all the columns: in this scenario high page 
sizes are favored, since they translate in fewer columns to be iterated through.
For \qro we observe a symmetric pattern.

When data has skew on the start times, instead, the patterns exhibited
by the two indexing orders are rather different.
First, we note that the gap between the best configurations of the two indexing orders is
much wider than with the other dataset.
Second, while the indexing first by duration exhibits a similar pattern on both datasets,
indexing first by start time performs best in the degenerate case of a single column , 
therefore with no partitioning of the start times at all.
This is a consequence of Observation~\ref{obs:dimension-order}. In particular,
having skewed start times exacerbates
the difference in selectivity between the range and duration constraints: when
start times are very concentrated on the timeline, the range constraint of a query is satisfied by either most of the intervals or by almost none.

Overall, we observe that indexing first by duration has either better or comparable performance compared to indexing by time first.
Therefore, we recommend to choose the former ordering of dimensions when building an \ours.

\section{Scalability with respect to the input size}
\label{sec:scalability}

\begin{figure*}
	\includegraphics[width=\textwidth]{images/scalability.png}
	\caption{Scalability of the index structures for the increasing dataset sizes.
		Synthetic datasets start from 10 million intervals up to one billion intervals.
		Each point corresponds to the best configuration of the given data structure for a particular dataset scale.
		The workload is of range-duration queries.
		\label{fig:scalability}}
	\Description{Scalability figure, showing that our proposed approach has superior performance for all tested
		dataset sizes, and scales gracefully.}
\end{figure*}

To test the scalability of the index structures, we consider datasets of increasing sizes.
For synthetic datasets we start with 10 million intervals, scaling by factors of 10 to one billion intervals.
For real-world datasets, we artificially increase their size as follows.
Given a dataset and a scale parameter $s$, let $\bar{t}$ the span of time covered by all the intervals in the dataset.
We make $s$ copies of each interval, and shift copy $i$ in time by $i\cdot \bar{t}$, for $i\in[0, s)$.
The underlying idea is to repeat the temporal patterns of the dataset on a longer time scale.

Figure~\ref{fig:scalability} reports the performance of the best configuration of each algorithm,
showing that the performance degrades gracefully as the dataset size increases, for all the data structures
involved in the comparison.
On synthetic datasets, for all tested sizes, \ours is about one order of magnitude faster than the
second-best data structure, \algo{grid-file}.

\begin{tcolorbox}
	\textbf{A note on my implementation of period-index-$\star$}.\\
	In~\cite{BehrendDGSVRK_SSTD19_period-index}, there is a note about how to save on
	empty cells, which are the vast majority in all the experiments we try (from 50\% up to 99\%).
	In the paper, there is the suggestion to use a linked structure, with non-empty buckets linked to one another.
	However, as I see it, this would imply that we lose the ability to index into the first non-empty bucket
	containing potentially overlapping intervals.
	Therefore, we would have to do a linear scan of these linked lists, which would absolutely kill performance.

	Given this, my implementation uses an array to hold all the cells, event the empty ones,
	thus allowing to directly index into it.
	Each cell, then holds a pointer to the next non-empty cell, allowing, at query time, to skip the empty cells.
	Still, we have problems of memory allocation, because we waste a great deal of space in empty cells.
\end{tcolorbox}
