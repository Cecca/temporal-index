\begin{abstract}
	Temporal information plays a crucial role in many database applications, however support for queries on such data is limited.
	We present an index structure to support queries of temporal intervals constraining both the range, i.e. the position along the timeline, and the duration of the interval.
	Our data structure features provable bounds on the running time of all the operations, and allows for a simple implementation, sports very predictable query performance.
	We benchmark our implementation on a variety of datasets and query workloads, investigating both the query rate and the behaviour of the individual queries.
	Our experimental evaluation shows that, compared to other data structures, our index indeed better supports range-duration queries.
	While other data structures are faster on specialized workloads constraining only the duration or the range, overall our data structure expresses the best compromise across all tested workloads.
\end{abstract}

\section{Introduction}

Temporal information plays a crucial role in many database applications: many database management systems provide indeed automated version control of the data and \emph{time travel} facilities, allowing to access past history efficiently.
In particular, past research concentrated on the important operators of temporal aggregation~\cite{DBLP:conf/icde/KlineS95,DBLP:journals/tkde/MoonLI03,DBLP:conf/edbt/BohlenGJ06,DBLP:conf/ssd/PiatovH17}, temporal joins~\cite{DBLP:conf/icde/PiatovHD16}, and time travel~\cite{DBLP:conf/sigmod/KaufmannMVFKFM13}.

All these approaches focus on handling temporal intervals, but concentrate on the \emph{position} of intervals along the timeline, ignoring another important attribute of intervals: their duration.

As a result, index structures to support more general selection queries with constraints on both the duration and the position in time of intervals have been missing~\cite{DBLP:journals/sigmod/KulkarniM12,DBLP:journals/tods/DignosBGJ16,DBLP:conf/ebiss/BohlenDGJ17} until recently~\cite{BehrendDGSVRK_SSTD19_period-index}.	

This paper takes as its starting point the observation that in several application both aspects of temporal information are useful to formuluate queries.
For instance, consider the following queries from different application scenarios:
\begin{itemize}
	\item[$Q1$:] \emph{"In a source code repository, find all the files that have not been modifyied for more than 2 years."}
	\item[$Q2$:] \emph{"Given a flight, find all the aircrafts that were flying at the same time."}
	\item[$Q3$:] \emph{"Find all the tourists who stayed in Bolzano between 2 and 4 weeks during August last year."}
\end{itemize}

The first query $Q1$ imposes constraints on the duration of matching events,
and we call it \emph{duration query}. In contrast, query $Q2$ retrieves
tuples based on the position of the events on the timeline; we call it
\emph{range query}. Finally, query $Q3$ constrains both types of information;
we call it \emph{range-duration query}.
Similar queries have been reported in
other application scenarios, e.g., air traffic
analysis~\cite{DBLP:conf/gis/SchullerBM10,DBLP:conf/adbis/BehrendMSW09,DBLP:conf/adbis/SchullerSB12}.

As shown in the above examples, time periods can comprise information about
their position on the timeline and the length of the period. However, index
structures over time intervals typically use only one of the two aspects. For
instance, the well-known relational interval
tree~\cite{DBLP:conf/vldb/KriegelPS00} is optimized for efficiently
determining temporal relationships between intervals but not interval
lengths. In the worst case, the entire index tree must be traversed if a
query solely contains restrictions on the interval length. 
Alternatively, the duration of the intervals can be indexed straighforwardly using a classic data structure such as a B-tree.
In this case, however, queries constraining only the range of the intervals will need to traverse the entire tree.

The two aforementioned indexes could be used at the same time, however combining different indexes is typically inefficient, since a query can have different selectivities in the two dimensions.

Therefore, to efficiently support queries constraining both the duration and the range of intervals we need to develop a data structure that can index both dimensions at the same time. A first step in this direction is \algo{Period-Index*}~\cite{BehrendDGSVRK_SSTD19_period-index}.

We improve on this previous work by proposing \ours{}, a two dimensional data structure that indexes both duration and position on the timeline at the same time.
In particular, our data structure partitions the intervals in a grid according to their start times and durations. Rather than being fixed, the boundaries between the cells of the grid are positioned by taking into account the data distribution, so to ensure that each cell contains the same number of intervals (with the exception of some edge cases, as we shall see in the following).
This allows obtain very predictable query times, which are proportional to the selectivity of the query.
The data structure also lends itself to a rather simple implementation, which we use to experimentally validate our approach using such implementation, finding that the overhead introduced by the data structure is indeed negligible, and that the running time in practice is largely proportional to the selectivity of the query.
This is in contrast to the other data structures we compare to.

Our contributions can then be summarized as follows:
\begin{itemize}
	\item We describe \ours{}, an index data structure that supports queries involving both the duration and range components of intervals;
	\item We prove bounds on its performance, which can be tuned with a single parameter;
	\item We provide an extensible open source implementation, which we benchmark against several baselines, showing significantly better performance across several workloads.
\end{itemize}

Related work can be split into two categories. First, index structures suitable for intervals, and second, index structures for two dimensional data. Due to the fact that time is constantly progressing, not all 2D indexes are suitable for temporal data. Finally, we also review structures similar to our approach that are used for temporal joins.

In the past, several data structures that support interval data have been proposed. One of the most popular structures is Edelsbrunner's Interval Tree~\cite{edelsbrunner80}, which is asymptotically optimal for temporal intervals and can be mapped into a relational database using B-tree indexes~\cite{DBLP:conf/vldb/KriegelPS00}. While the interval tree is efficient for range queries, it doesn't take into account the length of intervals.
Another index structure for intervals is the segment tree~\cite{deBerg00}, which builds disjoint segments (intervals) at the
leaf level using all start and end points in a relation. Internal
nodes merge all segments of their child nodes. A tuple is associated
with all sub-tree roots whose segment is completely covered by the
tuple's interval. Also this approach does not support queries that constrain the length of intervals.

Le et al.~\cite{DBLP:conf/sigmod/LeLTC13} propose a splitter for temporal and
multi-version databases with the aim to support load balancing for
distributed computing. For a user-defined number of buckets the goal is to
efficiently find the optimal splits by minimizing the number of intervals of
the largest bucket, whereas our goal is the computation of queries involving
position and duration of intervals in parallel.

Very recently, Behrend et al.~\cite{BehrendDGSVRK_SSTD19_period-index}
proposed an index expliticly supporting range-duration queries. Such index,
named \algo{Period-Index*}, partitions the time domain into \emph{buckets},
assigning an interval to all the buckets it intersects. Within each bucket,
intervals are further partitioned in \emph{levels} according to their
duration, with the minimum duration indexed within each level decreasing
geometrically. To support efficient indexing of times, each level is further
partitioned in the time domain. This data structure is adaptive to the
distribution of start times, while it assumes a Zipf-like distribution for
the durations of the intervals. As we shall see, our index structure removes
this assuption, supporting databases with arbitrary distributions of
durations.

Multidimensional indices can be used to index positions of intervals (using their structure or as points in the two-dimensional space) and their duration by using an additional dimension. R-trees~\cite{DBLP:conf/sigmod/BeckmannKSS90,DBLP:journals/talg/ArgeBHY08} are a variant of multidimensional indices that adopt object grouping using minimum bounding rectangles.
Quadtrees and octrees~\cite{DBLP:journals/acta/FinkelB74,Ulrich00,Samet05} are a different variant of multidimensional indices. The main idea of these trees is to recursively divide the space into disjoint partitions and to place objects into the best fitting partitions based on some criteria, such as fill factor of partitions or size of partitions.
The main issue of these approaches is that they suffer from additional dimensions, which substantially reduces their performance. 

A recent approach to multidimensional indexing is that of \emph{learned
indexes}~\cite{DBLP:conf/sigmod/KraskaBCDP18}: the proposition is that index
structures are \emph{models} mapping keys to records. Therefore machine
learning models can be used to provide this mapping, in lieu of the classic
data structures. In particular, Flood and
Tsunami~\cite{DBLP:conf/sigmod/NathanDAK20,
DBLP:journals/corr/abs-2006-13282} use Recursive Model
Index~\cite{DBLP:conf/sigmod/KraskaBCDP18} and a variant of decision trees to
model the position of records of the database, adapting to the distribution
of the data and of the query workload.

\section{Preliminaries}

We assume a linearly ordered, discrete time domain, $\domT$.  A
time interval is a set of contiguous time points, and $[\TS,\TE)$
denotes the closed-open interval of points from $\TS$ to $\TE$.  A
tuple in a temporal relation is timestamped with an interval that, for instance,
represents the tuple's valid time.  The schema of a temporal relation
is  given by $R = (\args{A}{m},\T)$, where $\args{A}{m}$ are the
nontemporal attributes with domains $\Omega_i$ and $\T$ is the
timestamp attribute with domain $\domT \times \domT$. A temporal
relation $\rr$ with schema $R$ is a finite set of tuples, where each
tuple has a value in the appropriate domain for each attribute in the
schema. We use $r.A_i$ to denote the value of attribute $A_i$ in tuple
$r$.  We use the abbreviations $\A = \{ \args{A}{m} \}$ and
$r.\A = (\args{r.A}{m})$, and we use $|\T|$ to denote the duration of time interval $\T$, i.e., $|\T| = \TE - \TS$.
Furthermore, given two intervals $T$ and $T'$, we denote with
$T \cap T'$ the set of time points shared by the two intervals, which, if
not empty, is itself an interval.

\subsection{Temporal queries}

Our index efficiently supports the three following types of queries (defined as in~\cite{BehrendDGSVRK_SSTD19_period-index}).

\begin{definition}[Range query]
	Given a temporal interval $T = [\TS, \TE)$ and a temporal relation $R$,
	a \emph{range query} $Q(R, T)$ is defined as
	\[
		Q(R, T) = \left\{r \in R : r.T \cap T \ne \emptyset \right\}
	\]
\end{definition}

\begin{definition}[Duration query]
	Given a duration interval $d = [d_{min}, d_{max}]$ and a temporal relation $R$,
	a \emph{duration query} $Q(R, d)$ is defined as
	\[
		Q(R, d) = \left\{r \in R : |r.T| \in [d_{min}, d_{max}] \right\}
	\]
\end{definition}

\begin{definition}[Range-duration query]
	Given a temporal interval $T = [\TS, \TE)$, a duration interval $d = [d_{min}, d_{max}]$,
	and a temporal relation $R$,
	a \emph{range-duration query} $Q(R, T, d)$ is defined as
	\[
		Q(R, T, d) = \left\{r \in R : r.T \cap T \ne \emptyset \wedge |r.T| \in [d_{min}, d_{max}] \right\}
	\]
\end{definition}

The latter type of query is a combination of the former two.

\section{Data structure}
\begin{figure}

An interval can be completely described by its duration and starting time, therefore
we can represent a set of temporal intervals as a set of points in a two dimensional 
space, with one coordinate being the duration and the other the start time.
A range-duration query with time interval $[\TS, \TE)$ and duration interval
$[d_{min}, d_{max}]$ in such a space is the polygon containing all the solutions,
which is defined by all points $(d, t)$ such that
\begin{equation*}
\begin{aligned}
  t < \TE \quad\wedge&\quad t < \TS - d \quad\wedge \\
          \quad\wedge&\quad d_{min} \le d \le d_{max}
\end{aligned}
\end{equation*}

\begin{example}
  Figure~\ref{fig:example-plane} lays out the points of table~\ref{tab:example}
  in the aforementioned space, along with the intervals associated to other flights
  on the same day.
  Note how the points (hence the intervals) are not uniformly distributed.
\end{example}

Our index is a grid, partitioning the tuples into disjoint buckets according
to the durations and the start times of their intervals.
The boundaries between cells are defined by using the empirical cumulative
distribution function of the duration and the starting times, so that
each cell contains the same number of intervals, which corresponds to the
\emph{page size} of our index.
This allows the data structure to adapt to the distribution of the input.

In the following we will focus on the interval attribute $T$ of relation $R$:
in the discussion we assume that each interval being inserted in the index
is associated with a reference to the tuple in $R$ it belongs to.

\subsection{Index construction}
\label{sec:index-construction}

\begin{algorithm}[t]
	\caption{\textsc{DefineBoundaries}$(I, b, k)$\label{alg:define-boundaries}}
	\KwIn{A collection of intervals $I$; a bucket size $b$; a \emph{key} function $k: I \to \domT$}
	\KwOut{An array of boundaries partitioning $I$ into up to $I/b$ buckets}

	Sort $I$ according to $k$, so that $I_i$ is the $i$-th interval in sorted order\nllabel{line:sorting}\;
	\Let{$B$}{$[]$}\;
	\Let{$i$}{$0$}\;
	\While{$i < |I|$}{
		\nllabel{line:iterate}
		\Let{$j$}{$i+b$}\;
		\If{$k(I_i) = k(I_{j})$} {
			\nllabel{line:heavy-bucket}
			\lWhile{$j < |I| \wedge k(I_i) = k(I_j)$} {
				\Let{$j$}{$j+1$}
			}
		} \ElseIf{$k(I_j) = k(I_{j+1})$} {
			\nllabel{line:split-bucket}
			\lWhile{$j > i \wedge k(I_j) = k(I_{i+b})$} {
				\Let{$j$}{$j-1$}
			}
		}
		Append $I_j$ to $B$\;
		\Let{$i$}{$j+1$}\;
	}
	\Return $B$\;
\end{algorithm}

Our grid structure partitions the dataset first along either the duration or start time dimension, and then along the other. The choice of which dimension to index first may of course impact the performance, depending on the query workload.
To make the exposition more concrete, in the following we will assume that the duration dimension is indexed first.
All the descriptions, considerations, and proofs also hold with the dimensions swapped.

Let $s$ be the \emph{page size} parameter.
The goal of the following construction is to ensure
that each cell of the grid contains at most $s$ intervals.
We divide the intervals into \emph{columns} based on their duration with two goals:
all the intervals with the same duration should be in the same column, and each column
should have at most $s^2$ intervals, unless there are more than $s^2$ intervals with the same duration.
In the latter case, all the intervals in the column shall have the same duration.
To this end, we invoke Algorithm~\ref{alg:define-boundaries} on the entire set of intervals,
setting the bucket size to $s^2$ and the key function $k$ to the duration.
In this invocation, we first sort all the intervals by increasing duration.
Runs of duplicate intervals are replaced by a single instance, equipped
with a list of references to the tuples timestamped with that interval.
Then, we iterativaly define bucket boundaries as follows.
We set the start of the first bucket to the duration of the first interval.
Then, we look ahead at the interval $s^2$ positions from it, and we can have one of three cases:
\begin{enumerate}
	\item the look ahead interval has the same duration as the one starting the column,
	      meaning that we are considering a run of more than $s^2$
	      intervals all with the same duration (line~\ref{line:heavy-bucket} of Algorithm~\ref{alg:define-boundaries}).
	      Then, the boundary of the bucket is set
	      to the current duration, and the start of the next column is set to the first interval
	      with duration strictly larger than the current one.
	\item the look ahead interval and the one immediately following it have the same duration,
	      which is however different from the one of the interval starting the column (line~\ref{line:split-bucket}
	      of Algorithm~\ref{alg:define-boundaries}).
	      In order not to break a run of same-duration intervals between different columns while
	      keeping the column itself smaller than $s^2$, we look for the first interval preceding the
	      look ahead with a strictly smaller duration, and make it the end of the column
	\item otherwise, the look ahead interval marks the end of the column.
\end{enumerate}

Then, each column is further partitioned according to the start time, invoking on it
Algorithm~\ref{alg:define-boundaries} with bucket size $s$ and key function $k$ set to extract
the start time of the intervals.

In each cell, the intervals are sorted by end time.

Along with the intervals in the grid, we store the boundaries between
the columns, as well as the boundaries between the cells in each
column, in sorted arrays.
This will allow, at query time, the efficient retrieval of the range
of cells containing intervals that match a given query.

\subsection{Querying the index}

Given a range-duration query $Q(R, \T, d)$ with $\T=[\TS, \TE)$ and $d=[d_min, d_max]$,
recall that a tuple $r \in R$ satisfies the query if the three following conditions
are met:
\begin{align}
	 & |r.\T| \in [d_{min}, d_{max}] \label{eq:duration-condition} \\
	 & \TS < r.\TE                  \label{eq:start-condition}     \\
	 & r.\TS < \TE                  \label{eq:end-condition}
\end{align}

First, find the two indices which identify the range of cells in the duration
dimension each containing at least one interval satisfying
condition~\eqref{eq:duration-condition} by means of binary search on the
boundaries of the columns. Let these two indices be $i_{start}$ and $i_{end}$.
Then, for each $i \in [i_{start}, i_{end}]$, let $[d_{min}^i, d_{max}^i]$
be the range of durations spanned by the $i$-th column.
By means of binary search, we look in the $i$-th column for the first cell
that contains the time $r.\TS - d_{max}^i$ and for the last cell
containing the time $r.\TE$.
In other words,
the start is the first cell such that the upper boundary on the start
time plus the maximum duration in the duration range of the column
is strictly less than the end time of the query

For each matching cell, we are in one of two cases:
\begin{enumerate}
	\item the cell's duration range is contained in the query's duration range.
	      In this case we iterate through just the intervals such that $\TS < r.\TE$,
	      leveraging the fact that intervals are sorted by end time, and we report them
	      as matches.
	\item otherwise all the intervals in the cell are evaluated against the query.
\end{enumerate}

\subsection{Updating the index}
\label{sec:updating-index}

Our index data structure can be extended to support both insertion and
removals of tuples. For a tuple $r$, with $r.t=(\TS, \TE)$,
by querying the index for the duration $|r.t|$ and the
start time $\TS$, we can identify the cell that should contain the tuple $r$.
If the interval $r.t$ is already indexed in the returned cell, then we simply append $r$ to its reference list.
Otherwise, inserting new intervals directly into cells might make them grow too large to be
able to make performance guarantees on the query time.
Therefore, we set a threshold $2s^2$ for \emph{light} columns and $2s$ for \emph{light} cells
(as we shall see in the next sections, we can bound the performance of \emph{heavy} cells and
columns even though they exceed the page size).
If a light column exceeds size $2s^2$ then we split the column,
if a light cell exceeds size $2s$ then we split the cell.

Splitting a column entails finding the median duration $\bar{d}$ among its intervals, and then replacing it
with two new columns with boundaries $[\bar{d}_{min}, \bar{d})$ and $[\bar{d},\bar{d}_{max})$, respectively.
Each of the two new columns requires the construction of new cell boundaries, which can be done by invoking
Algorithm~\ref{alg:define-boundaries}.

Similarly, to split a cell we replace it with two new cells with boundaries $[\bar{t}_{min}, \bar{t})$
and $[\bar{t}, \bar{t}_{max})$, where $\bar{t}$ is the median start time in the cell being split.
Then the intervals in each of the two newly created cells are sorted by end time.

In both cases, the boundaries are updated by inserting in order $\bar{d}$ (resp. $\bar{t}$) in the sorted
array of column (resp. cell) boundaries.

As for the removal of a tuple $r$ from the index,
we query the index to find the cell $C$ containing $r.t$, and we
remove $r$ from the reference list of the interval.
If this results in an empty reference list, then the interval is removed from the cell.
As a consequence the cell might be empty,
in which case case we remove the cell from the column, removing its lower boundary from the
array of boundaries unless it is the first cell, in which case we remove the upper boundary from the array.
This has the effect of making the neighboring cell \emph{cover} the span of times contained in the cell being removed.
Finally, if the removal of the cell results in an empty column, we remove the column, updating the boundaries array
in a similar fashion.

\subsection{Analysis}

We now provde guarantees on the time required by all the operations supported by our index structure.

\begin{theorem}
	Given a set of $n$ intervals, building the index requires time $\BO{n \log n}$.
\end{theorem}
\begin{proof}
	Computing the the bucket boundaries for the duration using Algorithm~\ref{alg:define-boundaries}
	requires time $\BO{n\log n}$ for sorting (line~\ref{line:sorting}) and time $\BO{n}$ for iterating through
	the sorted intervals to find the boundaries (line~\ref{line:iterate}): in fact, each of the $n$ intervals
	is visited at most once.

	Let $n_i$ be the number of intervals in the $i$-th column.
	Then, the invocation of Algorithm~\ref{alg:define-boundaries} to find cell boundaries
	requires time $\BO{n_i \log n_i}$, following the same argument as above.
	Therefore, the time for building the second level of the index is
	\[
		\BO{\sum_i n_i \log n_i}
		= \BO{\log n \cdot \sum_i n_i}
		= \BO{n \log n}
	\]
	and the theorem follows.
\end{proof}


\begin{theorem}
	Given an index over a set of $n$ intervals, with page size $s$, the time for
	answering a range-duration query is
	\[
		\BO{\frac{n}{s^2}\log\frac{n}{s} + \frac{n}{s} + s^2 + k}
	\]
	where $k$ is the number of intervals matching the query predicate.
\end{theorem}
\begin{proof}
	Let $\T = (\TS, \TE)$ be the query interval, and $[d_{min}, d_{max}]$ be the duration range of the query.
	Recall that the index construction algorithm may create a heavy column or cell when it cannot break down a group of intervals
	because they all share the same duration or start time.
	A range-duration query defines a \emph{query polygon} in the duration $\times$ start time space, as depicted in Figure~\ref{fig:example-grid}.
	For a given cell $C$, we denote by $k_C$ the number of intervals in $C$ matching the query.

	Before proceeding with the proof, we establish some fundamental facts about heavy columns and cells.
	\begin{itemize}
		\item \emph{All intervals in a heavy column have the same duration}.
		      If they didn't then we could split the column in two different columns
		      with fewer values.
		\item \emph{All intervals in a heavy cell have the same start time}.
		      If they didn't then we could split the cell in smaller cells.
		\item \emph{A heavy column does not contain heavy cells}. By the two
		      points above a heavy cell in a heavy column will contain intervals
		      with the same starting time and duration, i.e. copies of
			  the same intervals. 
			  However we store duplicate intervals only once (with a list of pointers to the related tuples, which are all either part of the output or not).
	\end{itemize}

	The query time is comprised of two parts: first we have to index into the grid to find the correct
	range of cells to query; second we have to filter the intervals in each cell to retrieve
	the ones that are part of the answer.

	As for finding the range of cells, we first have to find the first and last column by duration, which can be
	done by binary search over the $n/s^2$ boundaries between columns in $\BO{\log (n/s^2)}$ time.
	There can be $\BO{n / s^2}$ matching columns.
	In each such column, to find the range of cells satisfying the predicate we have to do a binary search
	over the $\BO{n/s}$ intervals in the column (since in the worst case we are inspecting a heavy column, which can have $\BO{n}$ intervals).
	Therefore, the overall time to find the correct cells is
	\[
		\BO{\log\frac{n}{s^2} + \frac{n}{s^2} \log\frac{n}{s}} =
		\BO{\frac{n}{s^2} \log\frac{n}{s}}
	\]

	Consider now the enumeration of the results from the cells. There are four possible cases for a cell $C$:
	\begin{enumerate}
		\item Both the cell's durations range and start times range fall within the query polygon.
		      In such cell, the query constraints on the duration and the start time are satisfied by construction.
		      The algorithm considers just the intervals of $C$ which are part of the output (since it iterates over
		      the cell's intervals by decreasing end time).
		      Therefore, the time spent enumerating intervals from the cell is $\BO{k_C}$.
		\item The cell's duration range satisfies the query constraint, but the starting time range doesn't.
		      If the duration colum containing the cell is heavy, then the cell cannot be heavy and the algorithm has to evaluate
		      $\BO{s}$ intervals.
		      Otherwise the cell might be heavy. In such case, all the start times are equal and satisfy the
			  query constraint (otherwise we will not be looking at the cell at all).
			  Therefore all the intervals in the cell match the query, and the time to enumerate them is $\BO{k_C}$.
		\item The cell's start times all satisfy the query constraints, but not all its durations do.
		      In this case we cannot be in a heavy column (which contains a single duration) and thus
		      the column containing the cell contains at most $s^2$ intervals.
		\item The cell doesn't completely satisfy neither constraint. In this case neither the column nor the cell can be heavy,
		      therefore the algorithm must only evaluate $\BO{s}$ intervals.
	\end{enumerate}
	All the cells of type (1) account for at most $\BO{k}$ in terms of running time.
	As for cells of type (2) there are at most $n/s^2$ of them. Overall they account for $\BO{\frac{n}{s}}$ interval evaluations.
	Cells of type (3) contain at most $s^2$ intervals. Therefore, overall they account for $\BO{s^2}$ interval evaluations.
	Finally is then a constant number of cells of type (4) (which are at the corners of the query polygon).
	These cells therefore account for at most $\BO{s}$ interval evaluations in total.
	Overall, the time to evaluate the intervals in the cells is
	\[
		\BO{\frac{n}{s} + s^2 + k}
	\]

	Combining this last result with the time to find the range of cells to query, we have that the overall query time is
	\[
		\BO{\frac{n}{s^2}\log\frac{n}{s} + \frac{n}{s} + s^2 + k}
	\]
\end{proof}

The above theorem exposes a fundamental tradeoff of our data structure:
using a smaller page size allows to improve the precision of the data structure
(by looking at fewer intervals that are not part of the query output), while
at the same time increasing the number of columns that need to be queried.

\begin{theorem}
	Inserting a tuple $r$ into the index requires
	time $\BO{\log n/s + n/s + s^2 \log s}$.
\end{theorem}
\begin{proof}
	Finding the cell that should contain $r.t$ requires time $\BO{\log\frac{n}{s^2}}$ to find the column, and
	time $\BO{\log\frac{n}{s}}$ to find the cell within the column (as in the worst case all the intervals
	in the index have the same duration).

	\cetodo{Define that splitting does not happen for \emph{heavy} columns or cells.}
	In such case, splitting a column with $2s^2 + 1$ intervals requires to find the median and
	to apply Algorithm~\ref{alg:define-boundaries} on the
	two newly defined columns, both operations that require time $\BO{s^2\log s^2}$.
	Furthermore, inserting the new column boundaries in order in the sorted array of boundaries requires
	$\BO{\frac{n}{s^2}}$ time.
	By similar reasoning, splitting a cell with $2s+1$ intervals requires time $\BO{s\log s}$,
	and updating cell boundaries by inserting in-order requires $\BO{\frac{n}{s}}$
\end{proof}

In a similar way we can prove the follwing theorem.

\begin{theorem}
	Removing a tuple $r$ from the index requires time $\BO{\log n/s + n/s}$
\end{theorem}

\label{sec:query-inspect}

		red dots are \qro queries, blue dots are \qdo queries.

We investigate the relationship between the selectivity of the queries (i.e. the fraction of the
input that satisfies them) and the time taken by different data structures to answer them.
Given that range-duration queries constrain both the time and the duration ranges,
for a query $q$ we can define the \emph{time selectivity} as the fraction of the input that
satisfies the time range constraint of $q$, and the \emph{duration selectivity} as the fraction
of the input satisfying the duration constraint of $q$.

For a given dataset, we build a query set such that queries are uniformly distributed on the
\emph{time}$\times$\emph{duration} selectivity plan: this way we have queries which are
very selective in only one dimension, or that are very selective in both dimensions, or that are not
selective at all.
The goal is to investigate the behaviour of each index structure for queries with different characteristics.
To account for the overhead of measuring time and to level out the effect of the CPU cache,
we measure the time taken to repeat a given query 100 consecutive times, which divided by 100
gives an estimate of the time required by the query.

Figure~\ref{fig:selectivity-grid} reports results for such a setup, with 1024 queries arranged in a 32$\times$32
grid, running on the \dataset{UZ Synthetic} dataset. Colors encode the time required by queries having
a particular combination of selectivity in the time and duration dimensions.
When reading Figure~\ref{fig:selectivity-grid}, remember that less selective queries require more time
just to iterate through the output.
Interestingly, different data structures exhibit different patterns in this plot, as a result
of how they access data.
The \algo{interval-tree} plot exhibits vertical bands: the data structure is only able to select intervals
based only on their position on the timeline, therefore for a fixed time selectivity of the query
the time employed to filter in the duration dimension does not depend on the selectivity of the query,
since all the candidate intervals need to be examined.
For similar reasons, \algo{b-tree} shows horizontal bands.
The data structures that explicitly index both dimensions, instead, tend to exhibit a more \emph{diagonal}
pattern, in particular \ours.
The pattern exhibited by \algo{period-index*} tends to be more similar to the \algo{interval-tree}.
This means that this index is more responsive to queries that are selective in the time dimension.
This is to be expected, since \algo{period-index*} is adaptive to the distribution of start times
in the dataset.
Its worse performance compared to \ours and \algo{grid-file} for queries of a given selectivity
is explained by the fact that some intervals might be represented multiple times in the index.

Figure~\ref{fig:time-vs-selectivity} reports the performance (in terms of query time) against
the \emph{overall} selectivity of the same queries.
Ideally, a data structure should answer the queries in time linearly dependent on the selectivity,
irrespective of whether the query is more selective in the time or duration dimension.
From the figure, we observe that \ours indeed shows the ideal behavior.
This is consistent with the theoretical analysis: the query time depends on the
output size plus some overhead proportional to the cell size, that can be amortized with
the correct parameter choice (which we will discuss in the next section).
Moreover, \ours is able to answer queries in a more timely fashion, compared to the other indices.

As for the \algo{grid-file}, the query times are more scattered: the efficiency of a query
in this case is in fact dependent on the density of the cells being considered.

Considering the \algo{b-tree} and \algo{interval-tree}, we have \qro and \qdo respectively
are answered with the worst performance, as we expect given that they are answered
by simple enumeration.
Other queries are answered with varying degrees of efficiency, with queries constraining
just the dimension on which the index specializes exhibiting a close-to-ideal behaviour.

Finally, \algo{period-index$\star$} answers queries with mixed performance, possibly as an effect of intervals
being replicated across several cells.
As we observed previously, queries restricting only the time are the ones answered most efficiently
by this data structure.

\subsection{Dependency on the page size parameter}
\label{sec:param-dep}

\begin{figure}
	\includegraphics[width=\columnwidth]{images/param_dependency.png}
	\caption{Dependency of the performance, in terms of queries per second, on the
		page size parameter.
		\label{fig:param-dependency}}
	\Description{}
\end{figure}

To test the dependency of the query time on the \emph{page size} parameter, we consider
a dataset with 10 million intervals with starting times both uniformly
distributed between 0 and 10 million and clustered around 10 random centers,
and durations distributed according to a
Zipf's law with $\beta = 1$.
As for the query workload, we consider batches of 20\,000 \qrd, \qro, and \qdo queries.

Figure~\ref{fig:param-dependency} reports the results of varying the page size of the
index under these different workloads, in terms of queries per second.
Interestingly, we observe different trends for different query workloads.
First, consider \qrd queries.
For both the distributions of start times that we consider, the peak performance
is around page size 100.
This behavior, with the performance increasing to a peak and then decreasing, is in line with
our complexity analysis of the query time.

If we move our attention, instead, to \qdo queries, we see
that increasing the page size is detrimental, with performance staying approximately the
same up until around page size 100.
This is also in line with our expectations: a larger page size means a coarser partitioning
of the dataset.
All the intervals in cells in the same duration column are considered as being part of the output,
therefore having less duration columns forces each query to evaluate more non-matching intervals,
thus increasing the running time.

Conversely, the performance improves with the page size for \qro queries.
In this case having fewer duration columns means that fewer groups of cells
will need to be queried, thus improving performance.

\section{Scalability with respect to the input size}
\label{sec:scalability}

\begin{figure*}
	\includegraphics[width=\textwidth]{images/scalability.png}

		Synthetic datasets start from 10 million intervals up to one billion intervals.
		Each point corresponds to the best configuration of the given data structure for a particular dataset scale.
		The workload is of range-duration queries.

To test the scalability of the index structures, we consider datasets of increasing sizes.
For synthetic datasets we start with 10 million intervals, scaling by factors of 10 to one billion intervals.
For real-world datasets, we artificially increase their size as follows.
Given a dataset and a scale parameter $s$, let $\bar{t}$ the span of time covered by all the intervals in the dataset.
We make $s$ copies of each interval, and shift copy $i$ in time by $i\cdot \bar{t}$, for $i\in[0, s)$.
The underlying idea is to repeat the temporal patterns of the dataset on a longer time scale.

Figure~\ref{fig:scalability} reports the performance of the best configuration of each algorithm,
showing that the performance degrades gracefully as the dataset size increases, for all the data structures
involved in the comparison.
On synthetic datasets, for all tested sizes, \ours is about one order of magnitude faster than the
second-best data structure, \algo{grid-file}.

\begin{tcolorbox}
	\textbf{A note on my implementation of period-index-$\star$}.\\
	In~\cite{BehrendDGSVRK_SSTD19_period-index}, there is a note about how to save on
	empty cells, which are the vast majority in all the experiments we try (from 50\% up to 99\%).
	In the paper, there is the suggestion to use a linked structure, with non-empty buckets linked to one another.
	However, as I see it, this would imply that we lose the ability to index into the first non-empty bucket
	containing potentially overlapping intervals.
	Therefore, we would have to do a linear scan of these linked lists, which would absolutely kill performance.

	Given this, my implementation uses an array to hold all the cells, event the empty ones,
	thus allowing to directly index into it.
	Each cell, then holds a pointer to the next non-empty cell, allowing, at query time, to skip the empty cells.
	Still, we have problems of memory allocation, because we waste a great deal of space in empty cells.
\end{tcolorbox}
